## 1. 题目理解与拆解

系统构成：8 台 CNC + 1 辆 RGV（沿直线轨道运行，双爪机械手 + 自带清洗槽）+ 上料/下料两条传送带。

三种情形：
1. **一道工序**：任意物料可在任一台 CNC 上加工完成，8 台 CNC 地位对等；
2. **两道工序**：物料须先经过"第一道工序 CNC"，再经过"第二道工序 CNC"，两类机床数量如何分配
   是模型要回答的问题之一；
3. **故障**：CNC 加工过程中约 1% 概率发生故障，人工排除耗时 10~20 分钟，未完成物料报废，排除后
   立即可重新接单；情形3要求对情形1、2都重新考察。

任务拆解为两层：
- **任务1（一般模型）**：给出对任意参数都适用的 RGV 动态调度模型 + 求解算法（不依赖具体数值）；
- **任务2（数值检验）**：用表1的3组参数分别跑通情形1/2/3，输出调度策略与系统作业效率。

这本质上是一个 **单资源（RGV）多站点（8 CNC）的实时排产/物料搬运调度问题**，决策变量是"RGV
在每个时刻该去服务哪台机床、执行装料还是卸料"，目标是最大化 8 小时班次内的合格产出数量（等价于
最小化 RGV 与机床的空闲浪费）。这是经典的 **在线动态调度（online dispatching）** 问题，而不是
一次性可以求出闭式解的静态规划问题——因为班次内 CNC 完工时刻依赖于之前所有调度决策，且情形3
还引入了随机故障，这决定了我后面选择的建模路线。

## 2. 候选建模思路对比

### 思路 A（放弃）：把整个 8 小时班次写成一个大型混合整数规划 / 排班表

设想：把 8 小时切成若干"事件时刻"，为每次 RGV 动作、每台 CNC 的每次装夹建立 0-1 决策变量，
一次性求解一个 MILP，得到全局最优的调度表。

**放弃理由**：
- 8 小时里 RGV 大约要执行 200~400 次动作（后面仿真验证了这个量级），变量与约束规模是几百到上千
  个"事件槽"×8 台机床，属于典型的 **NP-hard 车间调度/AGV 路径问题**，通用求解器在这个规模下无法
  在合理时间内给出可验证的全局最优解，也难以在论文篇幅内清楚地写出完整的 MILP 公式。
- 更关键的是，情形3引入了**随机故障**，故障发生时刻和恢复时长是随机变量，MILP 这种确定性静态规划
  框架天然不适合处理"事先不知道哪台机床何时坏、坏多久"的动态重调度问题——出了故障后需要实时
  重新决策，而不是提前把整张表算好。
- 题目也明确要求"给出RGV动态调度模型和相应的求解算法"，用词是"动态"，暗示官方期望的是一种
  **在线决策规则 + 算法**，而不是一次性求解的静态排班表。

这个思路我没有彻底放弃其"整数规划"的精神，而是保留了其中"目标函数=系统作业效率最大化"的
表述方式，把它写进了论文的目标函数定义里，但求解手段换成了下面的思路 B。

### 思路 B（采纳）：离散事件仿真 + 在线贪婪派工规则（dispatching rule）

把系统建成一个离散事件系统：RGV 每次完成一个动作后，"睁眼看世界"——扫描当前所有机床状态
（空闲待装料 / 加工中 / 完工待取料 / 故障中），根据一条**实时派工规则**选出下一步该做什么，
执行后推进仿真时钟，循环直到 8 小时结束。

**采纳理由**：
- 完全贴合"动态调度"的题意：规则本身对故障、随机性天然免疫——故障发生只是让某台机床的状态
  多了一种（'idle_empty'，因报废重新可用），派工规则不需要特殊改写就能应对；
- 可以在同一套仿真框架里，把"求解算法"具体化为若干条可比较、可复现的候选规则，通过实测选出
  更优的一条，这正是国赛官方强调的"讲清楚为什么选这个模型"的要求；
- 计算成本低，3组参数 × 多种情形 × 故障蒙特卡洛复制，全部可以在几秒到几十秒内跑完，也方便做
  充分的敏感性分析。

**代价（如实记录）**：仿真+贪婪规则不保证全局最优，只是一个可行的、可验证优劣的启发式解，这一点
我在论文"模型评价"里明确写清楚，不夸大。

### 思路 C（辅助，未单独成模型）：排队论近似解析估计

考虑过用 M/G/1 或者更简单的"生产节拍"近似公式，快速估计一个量级上限，比如
`理论上限吞吐量 ≈ min(8台CNC产能之和, RGV可服务能力上限)`，用来做仿真结果的合理性交叉检验，
但排队论的解析式无法刻画 RGV 在 4 个物理站位之间往返的**空间行程结构**（不同机床之间移动时间
不同、双爪机械手的携带约束等），所以没有单独作为主模型，而是保留作为"结果合理性检验"的一把
标尺（见第 6 节）。

## 3. 关键假设及论证

在写代码前，先把物理系统的"含糊之处"用假设显式钉死，逐条给出理由：

1. **原料无限供给假设**：上料传送带始终有毛坯可取。
   理由：题目没有给出毛坯到达速率或库存上限，8 小时连续作业场景下传送带断料属于生产管理问题而
   非本题调度问题的核心，国内已发表的多篇同题解法也采用此假设；若传送带真的会断料，那是一个
   完全独立的"来料计划"问题，超出本题范围。

2. **站位几何简化**：8 台 CNC 两两一组分布在轨道旁 4 个"站位"（1,2 号在站位1；3,4 号站位2；
   5,6 号站位3；7,8 号站位4），相邻站位间距为"1个单位"，这与题目给出的"移动1/2/3个单位时间"
   刚好对应轨道上最多 3 个单位间距（4个站位、最大间距=3），数据自洽，因此采用这一常见的官方图
   （原文附件1未提供，但表1参数与该几何结构完全吻合，是被广泛采用的复原方案）。

3. **上/下料点与站位1重合**：即物料进出口设在轨道一端（站位1附近）。
   理由：题目原文只给出"移动1/2/3单位"三档时间，没有第4档，说明轨道另一端到最远站位的往返
   都可以由这三档组合而成，若上下料点与站位1重合，从站位1到任意站位的直达距离恰好落在1~3单位
   区间内，与给定数据完全吻合，是唯一使数据不缺项的自然复原方式。

4. **上下料时间的取值口径**：表1"RGV为CNCx一次上下料所需时间"，我把它解释为**单次装料动作**
   或**单次卸料动作各自**耗费该时长（而不是"卸料+装料合计"共用一次该时长）。
   理由：原文经 doc→md 抽取后格式已知有丢失/错位（文件开头已声明），原句"一次上下料"存在歧义；
   若解释为"卸料+装料合计只需一次该时长"，则相当于卸料动作免费，与"机械手需要真实抓取动作"
   的物理直觉不符；采用"各自耗时"的解释更保守、更符合多数公开解法的处理方式，我在论文里明确
   声明这是对歧义表述的一种合理化选择，不隐瞒。

5. **清洗只发生在最终产品下线前**：一道工序的物料完工后洗一次；两道工序物料只在**第二道工序**
   完工后洗一次（半成品从第一道转到第二道不清洗）。
   理由：清洗的物理意义应是"清除切削液/铁屑，交付前的最终处理"，对中间半成品清洗没有工艺必要性，
   题目原文"物料清洗作业"也只提及一次性动作，未提示需要洗两次。

6. **下料点交货免建模**：物料被 RGV 从机床上取下并完成清洗（若需要）后即计为"下线合格品"，不再
   单独为"运回下料传送带"建模一段行程。
   理由：这是为了聚焦本题的核心决策难点——**RGV该先去哪台机床**，而不是让"运输到下料点"这一
   与调度决策关系不大的固定环节引入额外的状态空间（双爪占用记账）。这是一个**主动的简化**，
   我在"模型评价"里明确列为局限之一：真实系统里这段行程会占用 RGV 部分时间，若照原样建模，
   实际系统效率会比我们估计值略低。

7. **RGV 双爪能力的简化**：模型只显式维护"一个输入槽"（用于两道工序场景下搬运半成品），毛坯的
   抓取被视为"顺路免费获取"，融合进单次动作的行程里。
   理由：完全通用地对两个机械手爪做独立状态机建模会显著增加实现复杂度，且对"选哪台机床服务"这
   一核心决策的影响是二阶的；简化后仍然完整保留了"两道工序时下料点可以少跑一趟"这一双爪机械手
   带来的核心效率优势（半成品可以直接从第一道工序机床搬到第二道工序机床，不必绕回上料点）。

8. **故障发生时机的解释**："故障概率约1%"解释为**每次装夹加工一次物料时，该次加工过程中有1%
   概率发生故障**（而不是每单位时间1%概率）。
   理由：题目原文"故障的发生概率约为1%"紧跟在"CNC在加工过程中可能发生故障"之后，最自然的
   单位是"每次装夹加工"这个事件，而非无量纲的时间概率密度（原文未给出时间尺度，也没有说明是
   "每小时1%"还是"每分钟1%"，若按时间尺度理解会引入额外的不可考证的假设）。

9. **故障发生的具体时刻**：在该次加工时长内均匀分布（0~加工时长之间的某个随机点发生故障），
   故障维修时长在 10~20 分钟内均匀分布。
   理由：题目未给出更细的分布信息，均匀分布是信息量最少、最保守的中性假设，也便于蒙特卡洛复现。

10. **调度规则的比较标准**：以"班次内合格品完工数量"作为衡量各调度规则优劣的唯一指标（等价于
    题目要求的"系统的作业效率"），不引入额外的加权目标。
    理由：题目原句直接问"系统的作业效率"，完工数量是这个概念最直接、最没有歧义的量化对应物。

## 4. 模型骨架（详细公式见正式论文）

- 状态：8 台 CNC 各自的 `(角色, 站位, 奇偶, 状态, 剩余占用截止时刻)`；RGV 的 `(当前站位, 是否携带半成品)`。
- 决策时刻：RGV 每次完成一次动作后的时刻（事件驱动，不做定步长离散化，保证时间精度=原始秒级参数）。
- 决策空间：对所有"完工待取料"和"空闲待装料"的机床分别构造候选动作（卸料 / 装料 / 卸料+装料
  组合 / 半成品转运），计算每个候选动作的"行程+服务时间"成本。
- 派工规则（候选 3 条，见第5节实测对比后选定）：
  - 规则 A（分层：先处理卸料类动作，同层内选最近）
  - 规则 B（不分层，纯粹选择成本最小的动作 —— **最终采用**）
  - 规则 C（"产出密度"最大化：预计产出增益 / 动作耗时）
- 两道工序时的机床角色分配 `n1`（第一道工序机床数）作为设计变量，通过枚举 `n1=1..7` 仿真搜索最优。

### 5.1 仿真引擎首次自检

一道工序、三组参数，规则A：

```
$ python3 rgv_sim.py
```

RGV 利用率接近 100%，说明在一道工序场景下 RGV（而非 8 台 CNC）是系统瓶颈，这与许多已发表的
同题分析结论方向一致，是一个初步的合理性信号。

（此次搜索使用的是规则A，后续发现规则A不是最优规则，第5.3节切换规则后重新搜索了一遍，见下。）

用一个简单的"产能平衡"解析估算做交叉检验：若不考虑 RGV 行程，仅按两道工序加工时长配平，
`n1/n2 ≈ T_step1/T_step2`：
- 第1组：400/378≈1.06 → n1≈4，**与仿真搜索结果(4)吻合**；
- 第2组：280/500≈0.56 → 解析估算 n1≈3，仿真给出 n1=2，相差1台；
- 第3组：455/182≈2.5 → 解析估算 n1≈6，仿真给出 n1=3，**明显偏离解析估算**。

第3组这个偏离一开始让我怀疑代码有 bug，专门检查了一遍 —— 后来确认这是**合理的**，不是错误：
解析估算只考虑了"加工时长配平"，完全没考虑 RGV 在轨道上往返的空间成本；而我的角色分配是按
CNC 编号顺序切分（前 n1 台是第一道工序），n1 越大，第一道工序机床在空间上越分散，RGV 服务它们
的行程成本结构会发生非线性变化。也就是说，**两道工序场景下的最优机床配比，不是单纯由两道工序
加工时长的比例决定的，空间布局同样起作用**——这是仿真相对解析近似给出的一个有价值的额外发现，
我把它写进了论文的"结果分析"里，而不是当作 bug 悄悄压掉。

### 5.3 派工规则对比实验（关键的"调整"节点）

最初我的直觉是"应该优先处理完工待取料的机床（分层规则A），因为让机床空转等待是最大的浪费"。
实测后发现**这个直觉是错的**：

```
== 1-step, compare rules ==
[1, ('A', 189, 1.0), ('B', 221, 0.998), ('C', 212, 1.0)]
[2, ('A', 162, 0.997), ('B', 196, 1.0), ('C', 186, 0.998)]
[3, ('A', 189, 0.998), ('B', 230, 0.998), ('C', 218, 0.999)]
```

规则B（不分层、纯粹选成本最小的动作）在一道工序全部3组参数上都明显优于规则A（总产出
647 vs 540，高出约 20%）。原因是：分层规则A有时会强迫 RGV 舍近求远——放着近在咫尺、可以
立刻装料的空闲机床不管，跑去很远的地方给一台完工机床卸料，纯粹为了满足"卸料优先"这条硬性
规则，反而拉低了整体吞吐。这是一次**"直觉 vs 实测"发生冲突、以实测为准**的真实调整。

两道工序场景下重新对比三条规则（用重新搜索出的最优 n1）：

```
== 派工规则对比：两道工序（各组用其最优 n1）==
[1, 'n1=', 3, ('A', 118), ('B', 133), ('C', 136)]
[2, 'n1=', 2, ('A', 109), ('B', 118), ('C', 129)]
[3, 'n1=', 3, ('A', 133), ('B', 135), ('C', 134)]
合计: {'A': 360, 'B': 386, 'C': 399}
```

这里规则C（产出密度）总体上又略优于规则B（399 vs 386，约高3.4%），尤其第2组差距明显
（129 vs 118）。规则A在两种场景下都是最差的，可以放弃。B 和 C 互有胜负但都显著优于 A。

**最终决定**：为了全篇论文口径统一、结论可复现，主结果统一采用**规则B**（在所有测试场景中
从未明显落后，逻辑也最简单、最容易在论文里讲清楚：本质是"贪婪最近可行动作"、可以类比在线
最短作业优先 SPT），但在论文"模型评价"部分如实报告规则C在两道工序场景下平均还能再提升
约3~9%的空间，作为改进方向，不把这个发现藏起来。

同时，在规则B下重新做了一次 n1 搜索核对：

```
group 1 [(1,60),(2,115),(3,133),(4,129),(5,124),(6,112),(7,59)] -> best (3, 133)
group 2 [(1,78),(2,118),(3,114),(4,103),(5,106),(6,89),(7,47)]  -> best (2, 118)
group 3 [(1,55),(2,105),(3,135),(4,132),(5,128),(6,126),(7,99)] -> best (3, 135)
```
第1组的最优 n1 从"规则A下搜出的4"变成了"规则B下搜出的3"（129→133，产出更高）；第2、3组
最优 n1 不变。这提醒我：**最优机床配比是和派工规则耦合的**，不能脱离所用的调度算法单独讨论
"最优人力/设备配置"，这也写进了论文讨论部分。

### 5.4 故障场景：蒙特卡洛复现

发现故障场景引入随机性后，单次仿真结果会抖动，于是对每组参数、每种工序情形跑了 **40 次独立
蒙特卡洛复制**（不同随机种子），取均值和标准差：

```
=== 有故障场景 (1% 故障率, 10-20min维修, Rule B, 40次蒙特卡洛复制) ===
1 1step: mean=219.2 sd=2.3 | 2step(n1=3): mean=130.8 sd=2.2
2 1step: mean=193.7 sd=1.3 | 2step(n1=2): mean=115.5 sd=1.7
3 1step: mean=226.7 sd=2.8 | 2step(n1=3): mean=131.9 sd=2.1
```

与对应无故障场景（221/133、196/118、230/135）相比，故障造成的产出下降幅度约在 0.8%~3%
之间——考虑到故障率只有1%、且故障后机床能立即恢复作业，这个"小幅下降"符合直觉预期，是又一个
合理性检验通过的信号。

### 5.5 CNC 机床利用率（用于验证"RGV是瓶颈"这个结论）

一开始 `cnc_busy_time` 的统计代码写成了占位符 `cnc_busy_time[c.idx] += 0`（忘记补真实累加逻辑），
输出恒为0，及时发现并修复为按机床真实占用区间（含故障占用）累加：

```python
cnc_busy_time[c.idx] += c.busy_until - start_time[c.idx]
```
修复后重跑，得到（无故障，规则B）：

```
1 1step: avg_cnc_util=0.542   2step(n1=3): avg_cnc_util=0.458
2 1step: avg_cnc_util=0.493   2step(n1=2): avg_cnc_util=0.407
3 1step: avg_cnc_util=0.546   2step(n1=3): avg_cnc_util=0.380
```

机床平均利用率只有 38%~55%，而同一批次的 RGV 利用率接近 96%~100%，两者对照，**定量证实了
"RGV是系统瓶颈资源、CNC产能有富余"**这个此前只靠直觉判断的结论，是本题最重要的工程结论之一，
也直接支撑了论文"模型推广"部分——如果要提升系统效率，增加 RGV（或提升其移动/换料速度）比
增加 CNC 数量更有效。

### 5.6 敏感性分析

对第1组参数做了三类扰动实验（均基于规则B）：

**故障概率**（0~5%，一道/两道工序，40次MC均值）：
```
fault_prob=0.000  1step=221.0   2step=129.0
fault_prob=0.010  1step=218.9   2step=125.3
fault_prob=0.030  1step=213.7   2step=120.8
fault_prob=0.050  1step=210.3   2step=118.1
```
产出随故障率近似线性下降，两道工序对故障更敏感（斜率更大，因为一次报废在两道工序流程中
可能发生在任一阶段，且级联影响下游机床供料）。

**维修时长区间**（其余不变，fault_prob=0.01）：从(300,600)秒到(1200,2400)秒，一道工序产出
从219.7降到217.9，两道工序从125.3降到124.4——**维修时长的影响远小于故障发生概率的影响**
（在1%这个基准故障率下，维修时长每翻倍，产出只降约1%），这个量级关系合理：故障本来就是小概率
事件，故障"有没有发生"比"发生后修多久"对总产出的边际影响更大。

**RGV 移动速度缩放**（0.7~1.5倍，1.0为原始速度）：
```
speed_scale=0.70  1step=234  2step=139
speed_scale=1.00  1step=221  2step=129
speed_scale=1.50  1step=194  2step=111
```
RGV 变慢50%（scale=1.5），一道工序产出下降约12%，两道工序下降约14%——**验证了"RGV是瓶颈"
这个结论：只要动RGV移动速度这一个参数，系统产出就有清晰、单调、量级相当的响应**，而如果RGV
不是瓶颈，这种响应应该很微弱。这与5.5节机床利用率的发现相互印证，我把两者放在一起作为"结果
合理性检验"的核心证据链。

## 7. 自我批判：本解答的局限

1. **不是全局最优，只是一条经过实测比较的启发式规则**：规则B/C 是在3条候选规则里选出的相对
   较优者，没有做更大规模的规则搜索（如禁忌搜索、遗传算法调参的派工规则组合），也没有理论上的
   最优性证明。如果重来，我会再加入一条"预判未来完工时刻"的启发式规则（不仅看当前状态，也
   对2~3步之后即将完工的机床做预判排队），有可能进一步提升产出。
2. **双爪机械手的建模是简化版**：模型只显式追踪"一个半成品搬运槽"，完全省略了"卸料点回程"
   这段行程的显式建模（假设完工品下线免费交付），这会让模型给出的效率数值**略高于**真实系统
   （因为真实系统中，把成品送回下料点也需要占用 RGV 的时间）。如果重来，会把两个机械手爪都
   显式建模为独立槽位，代价是实现复杂度和调试时间显著增加。
3. **上下料时间的取值口径存在一定歧义**（第3节假设4），原文经格式转换后表述本身有歧义，我采用
   了偏保守的解释，但这确实是一个基于合理推断而非确凿依据的假设，在论文中已明确标注。
4. **两道工序机床角色分配采用"按编号顺序切分"，未做空间上更精细的交错布局搜索**：只测试了
   "前n1台做第一道工序"这一种分组方式，没有测试比如"奇偶交错分配角色"等更复杂的空间布局方案，
   5.2节已经发现空间布局对结果有非平凡影响，说明这里还有优化空间未被穷尽。
5. **附件1（系统组成流程图）、附件2（结果EXCEL模板）原文缺失**，本解答依据文字描述与参数表的
   自洽性重建了轨道几何布局假设（第3节假设2、3），这个重建虽然有数据自洽性支持，但严格来说
   仍是一种推测，不是100%确定还原了官方原图，这是本题在数据侧客观存在的限制，已在论文正文
   与假设中明确注明。
6. **故障分布假设（均匀分布）没有实际故障数据支撑**：题目只给出概率与维修时间区间，均匀分布是
   信息量最少的中性假设，如果有真实故障台账数据，应该替换为经验分布重新标定。
