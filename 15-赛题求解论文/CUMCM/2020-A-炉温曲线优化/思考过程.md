# 思考过程记录 —— CUMCM 2020 A题《炉温曲线》

> 本文件是解题过程的如实记录（研究日志），不是最终论文。所有代码片段与输出均为本次会话中 Bash 工具的真实执行结果，未经润色或编造。

## 1. 题目理解：这道题到底在问什么

先通读题目原文（`/Users/ephemeral/美赛/13-国赛CUMCM/真题2020/2020_A题.md`），拆解出几个关键信息层：

**(1) 物理系统**：回焊炉是一条"传送带 + 11个小温区（分属预热区/恒温区/回流区/冷却区4个大温区）+ 炉前/炉后区域"的一维管道。电路板（含焊接区域）匀速通过，各温区各自设定一个恒定的空气温度，板上某点（焊接区域中心）的温度是这个"外部温度环境"随时间变化后，板自身热响应的结果——本质是一个**运动物体在分段恒定/渐变外部温度场中的热传导-对流响应问题**。

**(2) 数据资产**：`附件.xlsx` 给出了一次真实实验的炉温曲线（709个点，t=19~373s，每0.5s一个采样点，速度70cm/min，温区设定175/195/235/255/25），这是我们**唯一能用来标定模型参数的真实观测数据**——必须充分利用，不能只是"建立模型"而不做参数估计。

**(3) 四个问题的递进结构**：
- 问题1：给定新的温区设定和速度，"复现"炉温曲线 → 考察正向仿真能力。
- 问题2：给定温区设定，反推"最大允许速度" → 考察在满足制程界限（表1的5条约束）下的单变量搜索。
- 问题3：在温区温度（±10℃范围内可调）和速度（65~100cm/min）联合可调的情况下，让"超过217℃的面积"最小 → 5维约束优化。
- 问题4：在问题3基础上再加一个"对称性"目标 → 多目标/加权优化。

这个递进结构说明：**核心是问题1的机理模型**，问题2-4都是在这个模型基础上做约束满足和优化，模型的准确性直接决定后面三问的可信度。所以我把大部分精力放在第一步：如何用真实数据把机理模型标定准，再复用这个模型去做后面的搜索和优化。

## 2. 候选建模思路

在写代码之前，我列了几种可能的建模路线，并逐一评估：

### 思路A："空气温度场 + 一阶集中参数(牛顿冷却)热响应" —— 最终选用

把焊接区域（0.15mm厚的薄层）当作一个热容极小、内部温度均匀的"点"（集中参数假设），它随传送带以恒定速度 v 运动，经过空间位置 x=v·t 处对应的空气温度 T_air(x)。板的温度 T(t) 满足一阶常微分方程：

dT/dt = k·(T_air(x(t)) − T(t))

这是牛顿冷却/加热定律的标准形式。k 是等效换热系数（无量纲化后单位是1/s），可以通过真实数据做最小二乘拟合得到。

**选择理由**：
- 物理上站得住脚：0.15mm的板/焊点热容极小，Biot数很小，可以合理假设board内部无温度梯度（集中参数假设成立）。
- 结构简单、参数少（只有k，或分heat/cool两个k），用真实数据能可靠标定，不容易过拟合。
- 计算代价低，可以在问题3/4的优化循环里被调用几千次而不会太慢——这对后续优化环节非常关键。

### 思路B：偏微分方程（一维/二维热传导方程，考虑板内部温度梯度）

把板当作有限厚度的固体，写出 ∂T/∂t = α∂²T/∂x² 加对流边界条件，用有限差分/有限元求解板内部（甚至沿传送带方向）的温度分布。

**放弃理由**：
- 板厚只有0.15mm，且题目只关心"焊接区域中心"这一个代表点的温度，板内部的温度梯度在这个厚度尺度下可以忽略（时间尺度上，0.15mm的热扩散时间常数是毫秒级，远小于炉内几百秒的过炉时间），引入PDE只会显著增加计算复杂度和待调参数，却换不来precision上的实质提升。这是典型的"过度建模"，不符合"先想清楚机理再选模型"的原则。

### 思路C：直接对真实炉温曲线做函数插值/曲线拟合（如样条、傅里叶级数），不涉及物理机理

把附件的573个点直接用样条拟合成一条光滑曲线，然后"猜测"新工况下曲线的形状做类比缩放。

**放弃理由**：
- 这是纯数据拟合，没有可迁移性——附件数据只对应一组固定的温区设定和速度（175/195/235/255，70cm/min）。问题1-4要求在**不同的温区设定和不同速度**下给出温度曲线，纯曲线拟合无法外推到新的输入组合，只能记住这一条曲线本身。必须要有一个把"温区设定+速度"映射到"温度曲线"的机理模型，曲线拟合不满足这个要求。

### 思路D（子选择）：空气温度剖面 T_air(x) 怎么建

这是思路A内部的一个关键子决策。题目原文明确提到："炉前区域、炉后区域以及小温区之间的间隙不做特殊的温度控制，其温度与相邻温区的温度有关，各温区边界附近的温度也可能受到相邻温区温度的影响"——这句话是在提示：**空气温度沿炉长方向不是理想阶梯函数，边界处有过渡**。我尝试了三个版本：

1. **纯阶梯**：每个温区内部恒温，间隙/炉前炉后直接按最近温区取值（无过渡）。
2. **宽过渡**：炉前/炉后整个25cm区间线性过渡（从室温渐变到温区1设定值）。
3. **窄过渡（最终选用）**：温区间5cm间隙内线性过渡；炉前/炉后区域绝大部分保持室温，只在与温区1/温区11相邻的最后5cm（与温区间间隙同宽）内做线性过渡。

第2版（宽过渡）用真实数据检验后效果最差（见第4节"遇到的问题与调整"），最终选择第3版。

## 3. 关键假设及其论证

1. **集中参数假设（板内无温度梯度）**：焊接区域厚度0.15mm，属于热容极小的薄层，可视为零维点。这是把PDE降阶为ODE的核心假设，理由见思路B的讨论。

2. **两个不同的换热系数 k_heat / k_cool**：加热区域（温区1-9，靠加热片对流+辐射）和冷却区域（温区10-11，靠风扇强制对流冷却）的物理换热机制不同，强制对流冷却的表面换热系数通常与自然/受迫热风加热不同，所以允许 k 在"空气温度高于板温（加热）"和"空气温度低于板温（冷却）"两种情形下取不同的值，而不是用同一个k。这是从数据拟合诊断出来的（用单一k拟合残差有明显的系统性正负号模式，见第4节），不是凭空假设。

3. **空气温度剖面：温区内恒定+边界5cm线性过渡**：温区之间有5cm的物理间隙（题目明确给出），间隙没有独立控温，其温度自然是相邻两温区的某种过渡，线性过渡是在没有更多信息时最简约的选择。炉前/炉后区域（各25cm）由于不直接受热源控制，绝大部分应接近车间环境温度25℃，只有紧邻温区1/温区11的边界附近会受到辐射/热对流泄漏的影响——所以让这个"边界效应"的空间尺度与温区间隙的5cm保持一致（而不是整个25cm都过渡），这个选择本身没有先验依据，但经真实数据检验（对比"宽过渡"与"窄过渡"两个版本的拟合优度）后被数据支持。

4. **t=0对应电路板前沿到达炉前区域入口（x=0），初始温度取车间环境温度25℃**：题目原文"电路板进入回焊炉开始计时"支持这个设定；真实数据里传感器在T=30℃时才开始记录（因为传感器阈值），这与我们模型里从25℃出发、约21秒后升到30℃的仿真结果吻合（见第4节验证），互相印证了t=0的定义是合理的。

5. **炉长几何**：11个小温区（每个30.5cm）+ 10个间隙（每个5cm）+ 炉前/炉后各25cm，总长 = 25+11×30.5+10×5+25 = 435.5cm。用真实数据反推：实验速度70cm/min=1.1667cm/s，真实数据记录到 t=373s 时结束（此时温度尚未回落到室温），373×1.1667=435.2cm，与435.5cm几乎完全吻合——这说明官方实验记录的时间窗口本身就是"进炉到出炉"，为我们的result.csv输出范围（覆盖到板完全出炉为止）提供了直接依据，而不是随意假设。

### 4.3 表1"制程界限"数值缺失问题的发现与解决（重要留痕）

用 Read 工具读取的 `2020_A题.md`（docx自动提取为纯文本的版本）里，表1"制程界限"的最低值/最高值单元格全是空的——一开始怀疑是文本提取时表格数字丢失，直接用记忆中"常见"的数值会违反"不编造数字"的原则，于是回到**原始docx的XML**里核实：

```python
import zipfile, re
z = zipfile.ZipFile('2020_A题.docx')
xml = z.read('word/document.xml').decode('utf-8')
cells = re.findall(r'<w:tc>.*?</w:tc>', table_xml, re.S)
for c in cells:
    texts = re.findall(r'<w:t[^>]*>([^<]*)</w:t>', c)
    print(repr(''.join(texts)))
```
结果：所有数值单元格文本都是空字符串 `''`。第一次核实确认"文本层确实是空的"，但这依然可能是因为数字是以**公式对象（OMML，Office Math Markup）**的形式插入表格，而不是普通文字——因为同一份文档里"±10℃"那句话也遇到了同样的问题（普通正则只抓 `<w:t>` 标签，抓不到公式里的 `<m:t>`）。检索发现：

```
可以进行<m:oMath>...<m:t>±</m:t>...<m:t>10</m:t>...</m:oMath>ºC范围内的调整
```

证实"±10℃"确实是官方原文给出的调整范围，只是extraction脚本没有解析公式对象。据此推测表1的数字也可能是用同样方式嵌入的，于是把正则改成同时抓取表格区域内的 `<m:t>` 标签：

```python
print('m:t in table:', re.findall(r'<m:t[^>]*>([^<]*)</m:t>', table_xml))
```
输出：
```
['0', '3', '-3', '0', '60', '120', '40', '90', '240', '250']
```
按行对应，得到表1的真实官方数值：

| 界限名称 | 最低值 | 最高值 | 单位 |
|---|---|---|---|
| 温度上升斜率 | 0 | 3 | ℃/s |
| 温度下降斜率 | -3 | 0 | ℃/s |
| 上升过程中150~190℃的时间 | 60 | 120 | s |
| 温度大于217℃的时间 | 40 | 90 | s |
| 峰值温度 | 240 | 250 | ℃ |

**这是本次解题过程中一个重要的自我纠错**：如果没有回到docx原始XML核实，很容易把这段当作"数据缺失"直接用记忆中的经验值替代，那就违反了"不编造数字"的纪律。这次的经验是：**docx转文本时，公式对象（OMML）里的数字不会出现在普通的`<w:t>`正则匹配里，遇到"看似空白"的表格/关键数值时，要检查是否有公式对象把它们"藏"起来了**。

### 4.4 用真实数据标定模型参数

先按"温区内恒温、间隙线性过渡、炉前/炉后区域整个25cm都线性过渡"的思路做了第一版模型，标定结果：

```
calibration result (k_heat, k_cool): [0.01701005 0.0055446 ] SSE= 59260.4
RMSE= 9.142   MAE= 6.828   max abs resid= 21.5 (at t=30.5s)   R2= 0.9698
```
R²=0.97看似不错，但残差在时间序列上有明显的系统性模式（先偏高后偏低），逐段打印发现：
```
t=  19.0 obs=  30.03 sim=  44.31 resid=  14.28
t=  29.0 obs=  43.08 sim=  64.46 resid=  21.38
...
t= 309.0 obs= 236.50 sim= 217.58 resid= -18.92
t= 319.0 obs= 224.25 sim= 207.19 resid= -17.06
```
说明模型在"刚进炉"阶段升温过快（因为整个25cm炉前区域都被建成了向175℃过渡的斜坡，实际炉前大部分应该还是室温），在"温区9到温区10的骤降"阶段又降得太快。于是尝试思路D的第3版（窄过渡：炉前/炉后绝大部分恒定室温，只在紧邻温区1/温区11的5cm内过渡），重新标定：

```
params (k_heat,k_cool,L≈0): [0.01905727 0.00584953]  
RMSE= 6.225   MAE= 4.396   R2= 0.986
```
拟合优度明显提升（R²从0.970提升到0.986，RMSE从9.14降到6.23）。同时做了一次网格实验，验证"空间平滑长度L"是否需要引入更宽的平滑核来解释边界效应：

```
L=  0.5  SSE=  23164.5
L=  5.0  SSE=  23460.9
L= 20.0  SSE=  25009.0
L= 60.0  SSE=  40190.2
```
SSE随L单调上升，说明真实数据并不支持"更大范围的空间平滑"这个假设，窄过渡（L≈0，相当于阶梯函数只在温区间隙5cm内做线性内插）已经是数据能支持的最简模型，不需要再引入额外的平滑参数（避免过度拟合，也符合奥卡姆剃刀）。

**这里有一个建模上的取舍**：最早尝试L=0.5cm这种极窄过渡时，整体拟合SSE最低（RMSE=5.72），比"5cm边界过渡"（RMSE=6.23）更好一点。但L→0意味着炉前区域到温区1之间存在一个数学上的"温度阶跃"，这会导致仿真出的"温度上升斜率"这个指标出现一个几乎不随传送带速度变化的、由阶跃直接决定的固定上限（约3.08℃/s，无论问题2里速度怎么变都几乎不变）——这在物理上不合理（阶跃越剧烈，瞬时斜率理论上应该趋于无穷，只是被离散化的dt限制住了，属于数值假象，而不是真实的物理限制）。既然表1里"温度上升斜率≤3℃/s"是问题2/3/4里要用来做速度可行域判断的关键约束，如果这个约束的数值对我们选的时间步长 dt 很敏感（数值假象），那用它来做决策就不可靠。所以我最终改用"5cm边界过渡"版本（拟合略差但更稳健、更符合"温区间隙都是5cm"的统一物理图像），牺牲了一点点拟合优度换取了约束判据在数值上的稳定性。这是一个**明确记录的建模取舍**，而不是简单地"哪个SSE小就用哪个"。

最终采用的模型参数：**k_heat ≈ 0.0191 /s，k_cool ≈ 0.00585 /s，边界过渡宽度=5cm，R²=0.986，RMSE=6.23℃，MAE=4.40℃**（对比整条曲线的温度跨度约30~242℃，这个误差量级是可以接受的）。

```
metrics at v*: rise=2.867 fall=-1.242 peak=239.998 t217=72.53 t150_190=107.80
```
即最大允许速度约为**67.9 cm/min**，此时其余四项约束都在安全区间内，唯一起作用（binding）的约束是峰值温度下限240℃。

### 4.7 问题3/4：约束优化

决策变量：温区1~5公共温度T15∈[165,185]、温区6温度T6∈[185,205]、温区7温度T7∈[225,245]、温区8~9公共温度T89∈[245,265]（均为实验基准值±10℃）、速度v∈[65,100]cm/min。用 `scipy.optimize.differential_evolution`（种群25×5维，最多200代）求解，目标函数=面积+罚函数（对5条制程界限的违反做二次罚项，权重5000）。

第一次尝试用 `workers=-1`（多进程并行）跑，报错：
```
RuntimeError: An attempt has been made to start a new process before the
current process has finished its bootstrapping phase...
```
这是macOS+Python 3.14下multiprocessing的spawn保护机制问题（脚本没有`if __name__=='__main__'`保护）。改成 `workers=1`（单进程）后又遇到：
```
AttributeError: module 'numpy' has no attribute 'trapz'. Did you mean: 'trace'?
```
新版NumPy已移除`np.trapz`，改用`np.trapezoid`后正常运行。单进程运行耗时约3.5分钟（200代×25×5维种群，每次目标函数评估约5-10ms）。

## 5. 遇到的问题与调整（汇总）

1. **表1数值"看似缺失"**：一开始以为是文本提取误差，差点要用"公开常见数值"顶上，后来通过检查原始docx的OMML公式对象找到了真实官方数值（0/3, -3/0, 60/120, 40/90, 240/250），避免了编造数字。
2. **单一k拟合残差有系统性模式**：诊断后改为heat/cool两个不同的换热系数，拟合质量显著提升。
3. **炉前/炉后区域该不该整体做25cm线性过渡**：整体过渡版本拟合明显更差（RMSE 9.14 vs 6.23+），改为"温区内恒温+5cm边界窄过渡"后拟合改善，且更符合"炉前/炉后大部分区域接近室温、只有边界受影响"的物理直觉。
4. **炉前/炉后边界过渡宽度w的选择存在"数据拟合最优"和"约束判据数值稳定性"之间的冲突**：w→0（纯阶梯，无过渡）拟合SSE最低，但会让"温度上升斜率"这个关键约束指标变成几乎不随速度变化的数值假象（阶跃导致瞬时斜率被离散化步长人为限制住，而不是真实物理极限）；最终选择w=5cm（与温区间隙宽度统一）而不是让SSE最小化的w≈0，这是主动放弃"数据上最优"换取"物理与决策上更合理"的选择，第4.8节(b)的敏感性分析进一步量化了这个选择对结论的影响幅度（v\*变化约3.5%）。
5. **`np.trapz`已在新版本NumPy中移除**：跑优化脚本时报 `AttributeError: module 'numpy' has no attribute 'trapz'`，改用 `np.trapezoid`。
6. **`differential_evolution(..., workers=-1)` 在脚本模式下触发multiprocessing的spawn保护机制报错**（macOS + Python 3.14，spawn start method要求`if __name__=='__main__'`保护），改为 `workers=1` 单进程后正常运行（换来的是优化耗时更长，但避免了额外的工程复杂度）。

## 6. 自我批判：这个解答的局限

1. **模型标定只用了一组实验数据（70cm/min，175/195/235/255/25）**。这意味着 k_heat、k_cool 是在这一个工况点上拟合出来的，外推到问题2-4里不同的温区温度、不同的速度时，本质上是在做"单点标定、多点外推"，外推距离越远（比如速度从70到100 cm/min，温度从175到185℃）可靠性就越低。如果有第二组、第三组不同工况下的真实实验数据，应该用来交叉验证k是否稳定，可惜本题只提供了一组附件数据。
2. **边界过渡宽度5cm是"物理直觉+对比实验"选出来的，不是唯一解**。更严谨的做法应该是把边界过渡宽度也作为待标定参数之一，和k_heat/k_cool一起用真实数据联合优化，而不是像本文这样先做几次离散网格对比再挑一个"看起来合理"的值。受限于时间，本文没有做这个联合标定。
3. **问题3/4的优化用差分进化（全局启发式算法）+ 罚函数处理约束**，不是精确的凸优化或KKT条件求解——好处是不需要目标函数可微、能处理离散化数值仿真带来的非光滑性，坏处是不能严格保证全局最优，只能说是"在合理计算预算内找到的一个较优解"，报告的最优值应视为一个可信的近似解而非理论最优解证明。
4. **问题4的"对称性"权重λ是经验取值**，论文里会做敏感性分析说明λ变化对解的影响，但λ本身没有一个"客观正确"的取值——这是本题目开放性的一部分（题目也没有给出对称性和面积孰轻孰重的量化标准），我们的处理方式是把这个主观权衡显式地作为一个可调参数呈现出来，而不是假装有一个客观唯一的答案。
5. **换热系数k是否应该进一步依赖温度本身（如辐射项应该是T⁴而不是线性)**：本文用的是线性牛顿冷却近似，如果炉温更高、辐射占比更大，线性近似的误差会增大；R²=0.986已经说明线性近似在本题温度范围（25~250℃左右）内是够用的，但这依然是一个简化。
6. **问题4的"对称性下限"结论依赖于本文选定的决策变量范围和固定的换热系数模型**：敏感性分析发现λ从0到30，不对称度都稳定在13.5s左右，我把这解读为"模型结构决定的下限"，但这个解读本身也是有前提的——如果k_heat/k_cool的标定值有误差（如局限1所述），这个"13.5s下限"的具体数值也会跟着变化，更严谨的说法应该是"在本文标定的模型下，问题3/4限定的决策变量范围内观测到一个相对稳定的不对称度下限"，而不是断言这是回焊炉工艺的普遍物理规律。
