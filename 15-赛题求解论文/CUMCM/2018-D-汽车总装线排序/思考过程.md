## 一、题目理解：这道题究竟在问什么

先用 Read 工具通读了 `/Users/ephemeral/美赛/13-国赛CUMCM/真题2018/2018_D题.md`。拆解后，问题的骨架是：

1. **对象**：一条总装线，每天匀速通过 460 辆车（白班/晚班各 230 辆），每辆车由品牌（A1/A2）、配置（B1-B6）、动力（汽油/柴油）、驱动（两驱/四驱）、颜色（9色）五个属性确定。
2. **决策变量**：这 460×7=3220 辆车在总装线上的**排列顺序**，以及总装完成后每辆车分流到 C1/C2 两条喷涂线中的**哪一条**。
3. **约束的层次并不相同**，这是本题的关键：
   - **强约束（不可违反，否则工艺/质量出问题）**：品牌顺序（先A1后A2，各占当日一半）；四驱/柴油连续≤2辆；颜色-喷涂线的硬性归属（蓝黄红只能C1，金只能C2）；颜色相邻的"允许表"（4(a)-(h)条）。
   - **区间型强约束**：黑色连续排列须在 [50,70] 之间，批次间隔（非黑色车辆数）≥20——这既有下限又有上限，比四驱/柴油的"只有上限"更难满足。
   - **软约束/成本型约束**：四驱、柴油的间隔"理想是≥10，5-9可接受但代价高"——题目原文明确给出了一个**分级代价函数**，不是简单的0/1约束；同配置尽量连续；喷涂线上颜色切换、尤其黑色与其他颜色切换代价很高。

这个结构立刻排除了"随手套用一个通用算法"的想法：这本质上是一个**带序列相关代价的车辆排序问题（Car Sequencing Problem, CSP的一个变体）**，工业界（ROADEF/PSA标致雪铁龙的经典赛题）早有成熟的问题范式——多种属性各自有"最大连续数/最小间隔数"的比例约束，目标是在满足硬约束前提下最小化软约束代价。这提示了两个方向：精确算法（约束规划/整数规划）与启发式构造+局部搜索。下面详细比较。

## 二、候选建模思路比较

### 思路A：整数规划/约束规划精确求解

把 3220 个位置、每个位置的车辆型号设为 0-1 变量，把"四驱连续≤2""黑色连续∈[50,70]"等写成线性约束，用 CP-SAT 或商业求解器求解。

**放弃理由**：
- 变量规模 3220（位置）×约 100+（型号组合）远超出常见 CP-SAT 在合理时间内可解的规模，尤其颜色相邻约束是"序列相关"约束，写成线性/CP形式会引入大量辅助变量（每个相邻位置对×颜色对）。
- 本地环境没有联网权限安装 `ortools`（后续验证确实 `pip install pandas` 都需要走 `--break-system-packages`/建虚拟环境，遑论大型求解器），如果引入会让"能否复现"变得不可控。
- 即使能建模，7天×3220辆车的规模下，商业级CP-SAT在无专业调参的情况下大概率也拿不到可行解证书，只会拿到一个不知道离最优多远的可行解——效果不见得比设计良好的启发式更好，但可解释性和可调试性远不如启发式。

### 思路B：贪心构造 + 打分函数（最终选用）

逐位置决策：维护"当前颜色游程""四驱/柴油游程与间隔计数""黑色批次目标"等状态，每一步从剩余车型池中选出**硬约束可行**的候选集合，再用一个**打分函数**（软约束代价+启发式偏好）选出代价最小的候选。

**选择理由**：
1. 题目本身把约束分成了"硬性禁止"和"代价很高但可以接受"两层，这正是贪心+代价函数最自然的建模方式——不需要伪造一个统一的0/1可行域，可以直接把题目语言（"5-9辆代价很高""间隔越多越好"）翻译成分段惩罚函数。
2. 计算量可控：3220个位置，每步候选数量在几十到一百左右，整体在秒级即可跑完，可以反复试验、调试、做敏感性分析（这在贪心/启发式上远比在精确求解器上容易）。
3. 工业界对这类车辆排序问题的主流解法本身也是"构造式启发式 + 局部搜索"（如 ROADEF 2005 挑战赛的获奖方法），选择这一路线并非"图省事"，而是与问题的真实求解范式一致。

### 思路C（子问题）：黑色批次的处理——三次迭代

黑色是最难啃的约束（下面第五节详细展开），围绕"如何决定何时开始/结束一个黑色批次"我实际上尝试了三种方案：
- **C1 全局单一批次计划**：先算一周黑色总量，用 `plan_batches` 切成若干个 [50,70] 的批次，运行时按顺序消费——**失败**（见下文"遇到的问题"）。
- **C2 打分偏置（scarcity/pacing/黑色让位分数）**：让黑色在打分函数里"该早点用就早点用，该让路就让路"——**部分缓解但没有解决根本问题**。
- **C3 按班次显式规划 A1→A2 合并批次 + 强制阶段顺序**：这是最终采用的方案，见第五节。

三次迭代不是"推翻重来"，而是从"让算法自己悟出规律"逐步收敛到"把已知的确定性信息（每个班次A1/A2各有多少黑色车）直接算出来，只把算法留给真正需要局部决策的部分（哪个具体型号、什么时候切换非黑色颜色）"。这是本题最大的经验教训，见后文自我批判。

## 四、模型建立

### 4.1 符号与状态

把每辆车抽象为四元组 `(drive, color, fuel, config)`（brand 由所处班次的位置隐式决定）。为了在贪心中判定硬约束和计算代价，维护如下运行时状态（对应代码 `new_state()`）：

- `last_color`, `color_run_len`：当前颜色游程。
- `black_run_active/len/target`：是否处于一个黑色批次中、已排了多少、目标长度是多少。
- `black_batch_queue`：**由离线规划器算好的、这个 A1 子块要依次开启的黑色批次目标长度队列**（见4.3）。
- `since_black_gap`：距离上一个黑色批次结束已经过了多少非黑色车（判定≥20间隔）。
- `last_drive/drive_run_len/since_4wd_gap`、`last_fuel/fuel_run_len/since_diesel_gap`：四驱、柴油的游程与间隔计数，逻辑与黑色对称但没有下限、只有"≤2辆"的上限。
- `last_config/last_config_brand`：仅在**同品牌**相邻车辆间比较配置是否切换。

### 4.2 颜色相邻图——一个必须显式做出的建模决策

题目 4(a)-(h) 条给出的是"允许搭配表"，(h)条声明"没有允许即为禁止"。逐条翻译成对称的允许对集合：

```python
ADJ_BASE = {
    ('黑','黑'),
    ('白','白'), ('白','蓝'), ('白','棕'),
    ('黄','银'), ('黄','灰'), ('黄','棕'), ('黄','金'),
    ('红','银'), ('红','灰'), ('红','棕'), ('红','金'),
    ('灰','灰'), ('灰','黄'), ('灰','红'), ('灰','金'),
    ('银','银'), ('银','黄'), ('银','红'), ('银','金'),
    ('棕','棕'), ('棕','黄'), ('棕','红'), ('棕','金'), ('棕','白'),
}
```

**关键的建模判断**：黑色在 (a) 条里没有被列为任何其他颜色的"允许搭档"，按 (h) 条字面意思，黑色似乎永远不能与任何其他颜色相邻——但这会导致黑色批次永远无法开始或结束，显然不是题目本意。规则 (b)-(g) 逐条读下来，每一条的措辞都是"颜色X可以与颜色Y间隔排列"，唯独 (a) 条对黑色只字未提它能与谁相邻，只规定了游程和间隔的**数量**约束。我的判断是：**(a) 条对黑色是自洽、穷尽的规定，只用游程/间隔约束管它，(h)条的"没有允许即禁止"针对的是 (b)-(g) 条里列出的非黑色颜色之间的搭配，不适用于黑色的边界过渡**。这是一个必须在论文里说明的主观判断，不是唯一可能的解读，但是能让模型自洽运行的解读。

代码里体现为：

```python
elif state['last_color'] == '黑' or color == '黑':
    pass  # 黑色可以与任何颜色在批次边界相邻
```

### 4.3 黑色批次规划——本题真正的难点

真实数据显示了一个此前没有预料到的结构（用 Bash 跑出来的）：

```
2018-09-17 白班 A1 black= 93 total= 182   (51%)
2018-09-17 白班 A2 black= 33 total= 48    (69%)
2018-09-21 白班 A1 black= 113 total= 188  (60%)
2018-09-21 白班 A2 black= 19  total= 42   (45%)
```

**A1 子块的黑色总量几乎总在 90-117 之间（超过70的上限，必须拆成≥2个批次），A2 子块的黑色总量几乎总在 17-43 之间（低于50的下限，单独一个子块永远凑不出一个合法批次）**。这不是算法的问题，是数据本身的结构：A1 占当天产量的大头（约79%），A2 占小头，而两者黑色比例相近，绝对量差距悬殊。

于是唯一可行的路径是：**A1 批次的"尾巴"要跟紧随其后的 A2 的黑色"合并"成一个批次**，因为品牌顺序固定是 A1→A2，两者在总装线上物理相邻。写了 `plan_shift_black(b1, b2)`：

```python
def plan_shift_black(b1, b2, lo=50, hi=70):
    lo_r = max(0, lo - b2)
    hi_r = min(b1, hi - b2)
    for r in range(lo_r, hi_r + 1):
        rest = plan_batches_strict(b1 - r)
        if rest is not None:
            return (r, rest)   # r=A1贡献给合并批次的尾量，rest=A1自身独立批次列表
    ... # 找不到时退化处理
```

对 7 天×2 班=14 个班次实际求解，**13/14 有精确解**（唯一失败的是 2018-09-19 晚班，B1=99,B2=43，99-r 落在 (70,100) 的"死区"里，无法整除成合法批次——这是数值上的巧合，不是算法缺陷，已在代码注释与论文里如实说明）。

### 4.4 三次踩坑与修正（如实记录）

**坑1：`split_shift` 重复计数**。最早把一天的型号数量拆成白班/晚班两半时，`shift2` 是用 `v - base1[k]`（即 `ceil(v/2)`）算的，而 `base1[k]` 后来又被"奇数余量"逻辑加了1分给 `shift1`——导致 `shift2` 把这个余量**又算了一遍**。实际运行发现晚班总数偏多（258辆而非230）：

```
白班 A1 182   白班 A2 48
晚班 A1 196   晚班 A2 62     # 196+62=258，而不是230！
```

修法：先确定最终的 `shift1`（含余量分配），再用 `shift2 = v - shift1[k]` 反算，而不是提前算好两份再叠加余量。修完后 7 天 14 个班次全部精确等于230/230。

**坑2：黑色邻接图漏掉了黑色本身**（第4.2节的判断在第一版代码里没有真正实现，只是"想到了但没写"）。第一版 `ADJ` 只含 `('黑','黑')` 这一条自环，遇到"从黑切换到其他颜色"或"从其他颜色切换到黑"时，硬约束过滤器判定为非法，被迫频繁降级（relax），一天之内触发 800 多次强制放宽——本质上黑色批次从未真正开始过。用 Bash 独立验证（不信任内部计数，重新扫一遍序列）发现问题后，才加上"黑色可以和任何颜色相邻"的规则，强制放宽次数从 794 降到 53。这是本题过程中最大的一次单点修复收益。

**坑3：黑色批次目标值设错**。合并批次的目标应该是 `reserve + b2`（A1贡献的尾量+A2的全部黑色），但最初代码只把 `reserve` 塞进了目标队列，导致批次在 A1 内部就"达标"结束（长度=reserve，如23），A2 后续的黑色成了另一个独立、同样不达标的批次。用 Bash 逐步打印 `black_run_target` 才发现这个一行代码的低级错误。

**坑4（未完全解决）：合并处仍偶尔有1-3辆车的"插花"**。即便目标值修对了，A1 尾部有时还会在黑色批次真正启动前，插入1辆本该更早排掉的非黑色车（因为该车在贪心的那一步找不到满足颜色邻接的非黑候选，被迫先用黑色顶上，之后黑色游程被这一辆"孤儿车"打断）。加了一条**硬性阶段锁**（A1子块：非黑色必须排完才允许开启黑色批次；A2子块：黑色必须排完才允许排非黑色）后有所缓解，但没有完全消除——这是数据+贪心局部决策的结构性矛盾，第六节会诚实报告其量级。

## 六、敏感性分析（真实运行结果）

### 6.1 随机种子稳定性

贪心在"打分并列"时用随机数破平局，换5个种子重跑：

```
seed= 100 black_runs=43 black_ok=17 fourwd_over2=1 diesel_over2=1 config_switch_rate=8.62%
```

**结论**：各项指标在5个种子间几乎不变（黑色合规批次16-17个、四驱/柴油超限恒为1次、配置切换率8.5-8.65%）。这说明当前的残余违规**不是随机性带来的运气问题，而是数据结构本身决定的**——换种子不会让 A2 子块的黑色总量从 33 变成 50。这个发现本身就是有价值的诊断结论，写进了论文的敏感性分析部分。

### 6.2 配置切换权重敏感性

```
config_switch weight=  3   → 切换率 8.80%
config_switch weight= 15   → 切换率 8.62%（默认值）
config_switch weight= 45   → 切换率 7.93%
config_switch weight=120   → 切换率 7.14%
```

权重提高40倍，切换率只从8.8%降到7.14%，说明"尽量连续"这条软约束已经接近贪心在满足其他硬约束前提下能达到的上限（继续加权边际收益递减）。

### 6.3 四驱间隔权重敏感性——一个"零敏感度"的诚实发现

```
gap权重×0.1  → 4WD间隔<10的比例 12.3%
gap权重×1.0  → 4WD间隔<10的比例 12.3%
gap权重×5.0  → 4WD间隔<10的比例 12.3%
```

权重变化50倍，间隔不合规比例**完全不变**。这说明剩余的13次间隔<10不是"软约束权重没给够"，而是被其他硬约束（品牌切换边界、颜色批次边界等）结构性地锁死了位置——加大权重无法解决，只有改变更上层的排产策略（如允许4WD批次跨越更大范围重新规划）才可能改善。这是本次敏感性分析中最有诊断价值的一条，如实写入论文。

## 八、自我批判：局限与重来会如何改进

1. **黑色批次的执行层仍有约60%的游程严格落在[50,70]之外**（合并视角下约79%），这是本解答最大的短板。根本原因是贪心的"逐位置局部决策"不具备完美的前瞻能力，即使有了`plan_shift_black`这样的离线全局规划，落到"每一步具体选哪辆车"时仍可能因为颜色邻接等其他约束在错误的时刻抢占了一个位置。如果重来，我会考虑把"黑色/非黑色的0-1标号序列"作为一个独立的、精确求解的子问题（这是一个规模小得多、只有两种"颜色"的一维批次调度问题，用动态规划或简单的区间调度即可精确求解），把它作为**硬约束提前锁定每个位置的黑白归属**，再让贪心只在"给定黑白归属"的前提下决定具体型号——这样能从根上避免"局部决策抢占位置"的问题，是我在本次时间预算内没有来得及重写的改进方向。
2. **打分函数的权重是手工设定的**，虽然做了敏感性分析，但没有做系统的网格搜索/贝叶斯优化去找"帕累托最优"的权重组合。对于一个训练示范解答，展示"权重敏感性分析方法论"比"调出全局最优权重"更重要，但如果是实际参赛，应该再花时间做更系统的调参。
3. **喷涂线分配是排完总装线之后的二阶段贪心**，没有和总装线排序联合优化，可能不是全局最优（总装线上为了配合黑色批次做的让步，可能在喷涂线上造成本可避免的切换代价）。联合优化是更完整的建模方向，但会显著增加复杂度，在本次时间预算内选择了分阶段处理。
4. **成本函数缺少真实的量纲**：题目要求"较低生产成本"，但没有给出具体的成本系数（比如换色一次economic cost是多少元），我们的打分权重是相对量纲，只能做"结构合理性"意义上的优化，不能宣称"成本最小"在货币意义上成立。这一点必须在论文里明确声明，避免过度宣称。
5. **黑色规则的"边界可以接任意颜色"是一个主观建模判断**（第4.2节），如果评委按更严格的字面解读（(h)条真的排除黑色的所有相邻），那么整个模型的可行域都会不同。应对方式是在论文里把这个假设显式列出来、给出理由，让读者/评委可以评估这个判断是否合理，而不是悄悄藏起来。
