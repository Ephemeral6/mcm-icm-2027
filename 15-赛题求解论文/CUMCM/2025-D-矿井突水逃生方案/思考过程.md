## 1. 题目理解与拆解

先通读题目原文（`/Users/ephemeral/美赛/13-国赛CUMCM/真题2025/2025_D题.md`），把四问拆成"是什么在变化、谁在动"：

- **物理对象**：矿井巷道网络，用图（端点=节点，巷道=边）表示；每段巷道是矩形断面（宽 4 m、高 3 m），用两端点连线代表。
- **水流规则**（题目直接给出，不是我们要建的黑箱）：
  1. 突水点以恒定流量 30 m³/min 涌出；
  2. 水流前锋向前推进时，锋后水位恒为初始水位 0.1 m（像一股"薄层front"在推进）；
  3. 遇到分叉节点时，只往"水平"和"下行"方向的巷道**平均分流**，不往"上行"方向流；
  4. "端点水流到达时刻"＝该点第一次见水的时刻；"巷道充满水时刻"＝巷道内水面达到最高点（即水深到 3 m）的时刻。
- **问题 1/3**：给定 1 个/2 个突水点，把整个网络里"谁、什么时候见水""哪些巷道、什么时候灌满"算出来。
- **问题 2/4**：给定突水通知延迟（1 分钟），结合水流模型，给矿工规划/调整最快且安全（水深 ≤0.3 m 才可涉水，超过绕行）的撤离路径。

这道题的核心难点不是"编程"，而是**把题目里几句简化的中文规则，翻译成一个自洽的、可在 977 条边、663 个节点的真实图上跑得动的数学模型**。规则本身没给出摩擦系数、曼宁公式、断面淹没面积-高度关系这些"标准水力学"参数，所以强行上完整的圣维南方程（Saint-Venant equations）既没有数据支撑，也偏离了命题者"给你一套简化规则，考察你能不能把它变成可计算模型"的本意。

## 2. 候选建模思路对比

### 候选 A：完整浅水波/明渠非恒定流数值模拟（Saint-Venant + 有限体积法）
这是"真专业"的矿井突水模拟方法（真实工程里用 FLOW-3D、TELEMAC 等做）。但：
- 需要曼宁糙率系数、断面形状随水位变化的过水面积/湿周关系、边界条件（出口水头）等题目**根本没给**的参数；
- 977 条边 × 时间步的显式/隐式求解，对一份"训练示范解"而言性价比很低，且极易在缺参数处退化成"拍脑袋赋值"，反而不诚实；
- 题目已经把水力学问题"降维"成了一句离散规则（平均分流 + 恒定初始水位），说明命题组期望的是**图论 + 质量守恒**层面的建模，而不是连续介质数值仿真。

**结论：弃用**，仅在"模型评价与推广"里作为进阶方向提及。

### 候选 B：等时间步"广度优先"扩散模型（忽略流量、只按巷道长度定速）
最省事的做法：不管流量分配，随便给一个"水流速度"常数（比如 1.25 m/s），BFS 式地按最短路径把各点"到达时刻"算出来，分叉时随便复制水流到每条分支。
- 问题：完全丢弃了题目明确给出的"突水量 30 m³/min""初始水位 0.1 m""向水平和下行巷道**平均分流**"这些定量约束——分流后每条支路水流应该变慢（携带的流量变小），而不是保持原速。这样算出来的"到达时刻"和物理意义脱节，无法用来做问题 3 的"两点汇流叠加"，也无法给问题 2 提供"水深"这个关键的可涉水判据。

**结论：弃用**，但它提醒我们：不管选哪个模型，都不能漏掉"水深"这个状态量，因为问题 2 明确按水深分档定走速。

### 候选 C（采纳）：基于质量守恒的"运动波前沿传播 + 局部积水灌满"模型
核心思想：把题目那句"以 0.1 m 初始水位向前漫延"翻译成一个连续性方程——前沿在某巷道内推进时，单位时间流入的体积 = 新湿润长度 × 断面宽 × 初始水深：

```
Q_e = B · h0 · v_front   =>   v_front = Q_e / (B · h0)
```

其中 `Q_e` 是分配到该巷道的流量（由分叉节点等分流规则逐级确定），`B=4 m`，`h0=0.1 m`。这样：
- 分流之后流量变小 → 前沿速度自动变慢，物理上自洽；
- 用 Dijkstra 式"最早到达时间"算法在图上传播，天然支持任意拓扑（含环、多分叉）；
- 到达之后如果巷道是"死胡同"（下游无法再往下/水平走，也不是出入口），water 没地方去，就会在原地"泡"起来，水深从 0.1 m 线性涨到 3 m（局部"浴缸"近似）——这样就有了"巷道充满水时刻"和"水深超过 0.3 m 不宜涉水"的判据。

选它的理由：**它是对题目文字规则最直接、最少臆造的数学翻译**，参数全部来自题目本身（B, h0, Q），没有引入任何题目未给的水力学常数；同时计算复杂度对 977 条边完全可控（数分钟内出结果），可以在真实数据上跑通。

### 问题 2/4 路径规划的候选比较
- **候选甲**：静态最短路（只按巷道长度，不考虑水），逃生方向直接 Dijkstra。× 完全没用上问题 1 的水流模型，也没法处理"顺水/逆水/超深绕行"。
- **候选乙（采纳）**：时间依赖最短路（time-dependent Dijkstra）。每条边的通过时间是"进入该边的时刻"的函数（干燥 4 m/s／顺水 2 m/s／逆水 1 m/s／深度>0.3 m 不可通行）。因为速度只会随时间"变差或不变"（干燥→变水→可能变深至不可通行），满足 FIFO（先进先出）性质——即越早出发/越早到达某点，后续绝不会更差——所以标准 Dijkstra（按到达时刻取最小堆）依然适用、依然最优，不需要更复杂的 Bellman-Ford 式重复松弛。这一点在写代码前专门推了一遍，确认没有"等一等水退了反而更快"的反例（本题水位只涨不退，天然满足单调性）。

## 3. 关键假设及其论证

1. **前沿推进速度只由质量守恒 `v=Q/(B h0)` 决定，与巷道坡度无关。**
   理由：题目没给曼宁糙率、坡度-流速关系，唯一可用的定量线索就是"初始水位不变 + 恒定流量"，这正好是一个连续性方程，不需要额外假设摩擦。代价：忽略了坡度越陡水流理论上越快的物理直觉——这是模型局限，后文会专门讨论。

2. **分叉节点处，把"来向"边排除后，对所有 `Δz ≤ 0`（水平或下行）的邻边做严格等分。**
   直接翻译题目原句"向水平巷道和下行巷道平均分流"。等分对象不含来向边，因为水不会流回上一段。

3. **出入口（红点）视为自由排水口：水流到达出入口后即认为流出矿井，不再继续向出入口下游的巷道传播、不参与"淹没-回灌"。**
   理由：出入口连接地表，物理上水会经此排走而不是继续在井下憋压倒灌；这是题目没明说但符合常识的边界条件，问题 2 里给出的出入口坐标经验证与网络中节点 `P0393`、`P0526` 坐标完全重合（见第 4 节代码输出），说明这两个点确实是网络里真实存在的、地位特殊的节点。

4. **巷道"是否会被灌满（terminal/backfilling）"由下游连通性递归判定：如果一条巷道下游（不含来向）没有任何非上行分支，或者它所有下游分支都已经被判定为"堵死"，则该巷道会被判定为局部积水、水位从 0.1 m 涨到 3 m；否则视为"过水巷道"，在本模型时间尺度内维持 0.1 m 薄层水流，不判定为"充满"。**
   理由：题目要求同时输出"到达时刻"和"充满时刻"两种量，说明命题者预期二者不同——多数巷道只是"水流经过"，只有死胡同/局部洼地才会真正"灌满"。完整地模拟"水位涨到顶了以后倒灌回上游巷道"需要对整个连通积水区做统一水位的库容-水位曲线仿真，这是一个可以做但明显更重的三维库容仿真问题（详见"模型评价"），本解答采用"局部浴缸"近似，并在论文里明确写清楚这是简化。

5. **工人在一条巷道内的水情（干/湿、水深、顺逆水方向）在其"踏上该巷道那一刻"被评估一次，并在穿越该巷道全程保持不变。**
   理由：工人步速（1~4 m/s）与前沿推进速度（未分流时 1.25 m/s）量级相当，巷道平均长度 128.5 m，工人穿越一条巷道平均只需 30 秒~2 分钟，而水情变化的时间尺度（到达时刻的差值、充满所需时间）往往是几分钟到几十分钟量级，所以"入边定状态"是一个二阶量级的近似，用于把问题转化成可解的时间依赖最短路问题，而不必对每条边做逐秒的动态耦合。

6. **两个突水点（问题3/4）在同一条边上"相遇"时，按流量可加（下游合流处流量取二者之和）处理充满时刻，到达时刻按"先到者为准"处理。**
   理由：到达时刻只关心"第一次见水"，先到的水前沿显然决定了该时刻，第二个源头即使后到也不会让"首次见水"更早；但灌满速度取决于净流入体积，两股来源汇合后流量应叠加，这是最直接的质量守恒扩展，但没有严格重新做全局的分流网络重算（即合流点下游的分流比没有用"A+B合并后的新流量"重新分配到更下游节点），这是问题3/4部分的一个已知简化，第 6 节会展开自我批判。

### 4.5 敏感性自检：验证质量守恒的自洽性

在写敏感性分析之前，先做了一个"模型正确性自检"：如果 `v=Q/(B h0)` 是对的，那么把 Q 翻倍，所有到达时刻应该精确减半；把 h0 翻倍，到达时刻也应该精确翻倍。实测（附件2，单源 A）：

```
Q=15  -> 平均到达时间 1220.44 min
Q=30  -> 平均到达时间  610.22 min   # 恰好是 Q=15 时的一半
Q=60  -> 平均到达时间  305.11 min   # 恰好是 Q=30 时的一半

h0=0.05 -> 平均到达时间 305.11 min
h0=0.10 -> 平均到达时间 610.22 min  # 恰好是 h0=0.05 时的 2 倍
h0=0.20 -> 平均到达时间 1220.44 min # 恰好是 h0=0.10 时的 2 倍
```

完全线性、精确成比例——说明代码对 `Q` 和 `h0` 的实现是自洽的，没有隐藏的单位错误或逻辑漏洞。这比"跑出一个看起来合理的数字"更能说明模型被正确实现了。

也就是说，虽然这些巷道**最终**都会见水，但工人经过的那一刻水还没到——工人比水先走一步，全程干燥速度通过。这不是巧合：`escape_route` 是按"通过时间最短"搜索的，天然会优先绕开会拖慢速度的湿巷道（如果有替代路径的话），这恰好验证了算法在"避险"这个目标上是有效的。

附件2 工人3 的路径里有 3 段边落在了已经被水前沿覆盖的区间内，实测平均速度依然接近 4 m/s（因为大部分路径仍是干燥的，湿的那几段被 2 m/s/1 m/s 拖慢但影响不大）。

### 4.7 问题3：双源叠加

第二个突水点延迟 4 分钟（附件1的B1）/ 5 分钟（附件2的B2）开始。做法：对 A、B 分别单独跑一次前沿传播（B 用自己的局部时钟，最后整体加上延迟偏移），逐节点取 `min(tA, tB+delay)` 作为综合到达时刻；对边，若被 A、B 都覆盖，充满时刻计算时把两支流量相加（`q_comb = qA+qB`），加速灌满。

实测：

```
附件1 Q3: 综合可达端点 663 / 977条巷道全覆盖（本来单源就已经全覆盖，B源主要是加速局部灌满）
附件2 Q3: 综合可达端点 41 (单源A时只有28)，可达巷道 55 (单源A时只有40)
```

第二个源头显著扩大了附件2 网络的受影响范围（28→41 个端点），说明 B2 点（40.37 m高程，比A2的6.33 m高得多）覆盖了网络里完全不同的一片区域——这与"两个突水点分处不同高程、各自向下游扩散"的直觉一致。

### 4.8 问题4：调整逃生路径

在 B 点突水后 1 分钟（附件1 是 t=5 min，附件2 是 t=6 min）发布调整方案。做法：先按问题2的原方案算出每个工人在 t_adj 时刻走到了原路径的哪一段（按已耗时间线性插值定位到某条边的某个分数位置），在该位置插入一个新的虚拟节点，从这个"工人此刻的真实位置"出发，用问题3算出的双源综合水情重新做一次时间依赖 Dijkstra。

实测：

```
附件1: 工人1 原计划15.88min→调整后16.45min，且改道从出入口P0393→P0526
       工人2 原计划12.17min→调整后12.38min，仍走P0393
       工人3 原计划18.10min→调整后18.50min，改道P0393→P0526
附件2: 工人1 原计划12.20min→调整后12.41min，仍走P0393
       工人2 原计划18.13min→调整后18.50min，改道P0393→P0526
       工人3 原计划16.18min→调整后16.75min，改道P0393→P0526
```

时间只多了 0.2~0.6 分钟（多数是因为要绕开第二突水点新淹没的巷道，改走另一个出入口），量级合理，没有出现"绕一大圈"的病态结果，说明调整算法在这两组数据上表现稳定。

### 4.9 用延迟通知做一次"逃生窗口"敏感性分析

额外做了一组分析：把"发布通知"的时刻从 0、1、3、5 拖到 10 分钟，看逃生总耗时怎么变（附件2，单源A情形）：

```
notify t=0  : 工人1 travel=11.20  工人2 travel=17.13  工人3 travel=14.46
notify t=1  : 工人1 travel=11.20  工人2 travel=17.13  工人3 travel=15.18
notify t=3  : 工人1 travel=11.20  工人2 travel=17.13  工人3 travel=20.17
notify t=5  : 工人1 travel=11.20  工人2 travel=17.13  工人3 travel=20.17
notify t=10 : 工人1 travel=11.20  工人2 travel=17.13  工人3 travel=20.58
```

工人1、2 的"行走耗时"始终不变（说明他们的最优路径全程不受水情影响，无论多晚出发都一样快），工人3 随着通知延迟增大要多花 5~6 分钟（因为原本更近的路径逐渐被淹没至 >0.3 m 而不可通行，被迫改道），这是一个很干净、可解释的"通知越晚、代价越大"的定量例子，写进了论文的敏感性分析部分。

## 5. 遇到的问题与调整（如实记录）

1. **9.4×10¹⁶ 分钟的天文数字**（见 4.3）——一度怀疑是代码 bug，排查后确认是模型在完全平坦网络下的真实数学行为，最终决定"如实呈现 + 限定有效报告窗口"而不是偷偷改规则或者藏起来不提。这是本次解题过程中最大的一次"模型-实现-物理直觉"三方对齐的调试。

2. **"巷道充满时刻"的物理含义反复推敲**——最初想直接对每条边独立套用"进水量=Q_e·t"算灌满时间，后来意识到大多数边其实是"过水"而非"憋水"，如果不区分死胡同和过水巷道，会把所有边都算出一个（其实没有实际意义的）"充满时刻"。加入了下游连通性的递归"堵死"判定后，附件1单源情形下 977 条边里只有 922 条被判定为会灌满，56 条维持薄层过水，这个数字分布本身就是模型自洽性的一个佐证。

3. **问题3双源合流点的流量处理**——一开始只做了到达时刻的 `min()` 合并，跑完才意识到"充满时刻"如果不处理流量叠加就会低估灌满速度（两股水都往同一条死胡同灌，理应比单源更快灌满）。补加了 `q_comb = qA+qB` 的处理。这是一个已知的近似（合流后没有重新对更下游的分叉重新分配流量比例），在论文"模型评价"里专门说明。

4. **result4 模板行的"最后 / 出入口坐标 / 序号"字样**——一开始没看懂模板最后一行的意图，用 openpyxl 把原始单元格值和合并单元格范围都打印出来后才确认：那只是模板作者标注"这里要填出入口坐标"的提示文字，并非需要保留的字面值，实际填表时应替换成真实的最终坐标/时间/巷道编号。

## 6. 自我批判：这版解答的局限

1. **坡度对流速的影响被完全忽略**——`v=Q/(B h0)` 只用了质量守恒，没有用到任何坡度/摩擦信息。现实中陡直巷道水流应该更快。之所以这样处理，是因为题目没给曼宁糙率之类的参数，强行编一个坡度-流速关系式反而是不诚实的臆造。如果重做，可以尝试引入一个"无量纲修正因子"，在坡度较大时按 Manning 公式的量级做定性加速，但需要额外假设糙率系数，会引入题目未给定的自由参数。

2. **"局部浴缸"灌满近似，没有做真正的全局回灌（backwater）仿真**——真实物理中，一段死胡同灌满以后，多出来的水会顶着往上游倒灌，可能把原本"过水"的巷道也变成"憋水"巷道，甚至绕开原本因地形被排除的"上行"巷道（水位足够高时可以漫过局部高点）。这是一个完整的三维连通库容-水位仿真问题，工作量远超本次时间预算，本解答只做了"死胡同独立灌满"的一阶近似，在论文里已明确标注为简化并给出改进方向。

3. **双源叠加时下游分流比没有用合并后的总流量重新计算**——只是在"两支流都覆盖到的边"上把流量相加，用于计算充满时刻，但并未往下游继续传播这个叠加后的更大流量、重新做一次分流。这会在多个突水点非常接近、且离出入口不远的情形下引入误差；本题两个突水点之间距离较远（附件1 A1-B1 直线距离约 2020 m），实际计算显示两片受影响区域交集有限，误差可控，但方法论上仍是一个已知简化。

4. **工人穿越单条巷道时的水情"入边定格"近似**——没有对水位/水深做逐秒的连续追踪，而是在进入某条巷道的瞬间评估一次水情并保持到走完整条巷道。已在第 3 节假设 5 论证过这是二阶量级的近似，但在个别"巷道很长 + 水位正处于快速上涨阶段"的边界情形下可能低估风险，理论上更严谨的做法是把巷道再切分成更细的子段来提高时间分辨率。

5. **50000 分钟报告窗口是一个人为设定的工程阈值，不是从模型内部推导出来的"自然截断点"**——直方图显示附件1 的到达时刻是连续跨越 17 个数量级分布的，没有天然的"合理/不合理"分界线，50000 分钟只是"比附件2 真实最大充满时刻略大、同时明显小于组合爆炸区间"的一个折衷选择，换一个阈值报告表里的空白格数量会变化。这一点在论文里已明确注明是人为设定，不是模型输出。

6. **没有跑完整的"全部663个端点+977条巷道"的详细表格审阅**，只对统计量（数量、最值、分布）做了抽样级核验；虽然导出的 result*.xlsx 文件是全量真实计算结果（已用 pandas/openpyxl 写出，未做任何截断编造），但受时间预算限制没有逐行人工复核每一个数字。
