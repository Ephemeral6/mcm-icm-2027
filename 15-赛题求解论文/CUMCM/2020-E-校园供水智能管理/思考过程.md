## 一、题目理解：这道题到底在问什么

先把 2020E 的原题（`/Users/ephemeral/美赛/13-国赛CUMCM/真题2020/2020_E题.md`）拆成五个子问题，逐条翻译成"数据上要做什么"：

| 原题问题 | 我的理解（数据/建模任务） |
|---|---|
| ① 统计分析各水表变化规律，给出不同功能区用水特征 | 需要把 93 个水表按"宿舍/教学楼/办公楼/食堂/…"贴标签，再看每类的时间序列形状（小时曲线、工作日/周末对比） |
| ② 结合水表层级关系建立"水表数据之间的关系模型"，用数据分析模型误差 | 附件"水表层级"给的是一棵树（一级→二级→三级→四级表计编码），逻辑上"母表读数 ≈ 子表读数之和"，要把这个关系用回归/水量平衡定量化，并给出误差指标 |
| ③ 分析管网漏损，题目提示"维护良好的网络约 5% 失水，老旧网络更多" | 用②中的母子表差值占比，作为漏损率的经验估计量，和 5% 基准比较 |
| ④ 从实时数据中定位暗漏发生的位置 | 漏损定位问题——不能只看总表，要下钻到树的每一层节点，找到"差值异常"的具体分支；对于树已经到叶子、无法再拆分的建筑内部，要用另一种方法（夜间最小流量法）做"最后一公里"定位 |
| ⑤ 结合水价和维修成本给出最优维修决策 | 题目本身没给水价和维修成本数据，明确说"你了解的水价和维修成本"——即允许/要求使用外部常识性数值，但必须清楚标注这是示例值，不是数据集给出的 |

一个容易踩的坑：题目字面是"漏损"，但真的把数据打开后会发现，"母表 - 子表和"这个差值，混杂了至少三种完全不同的物理含义：
1. **真实管网漏损**（地下暗漏、老化管道渗漏）——这是题目本意；
2. **子表并未穷尽母表下游所有用户**（比如母表覆盖一整片区域，子表只挑了其中几栋楼建了子表，还有别的用户没被列为"子节点"）——这不是漏损，是"账面缺口"；
3. **表具本身的计量误差/校准漂移，甚至拓扑关系本身就不完全对应实际管网连接**——这是"表的问题"不是"管网的问题"。

这三者必须先甄别清楚，否则会把"没装子表的用户"错当成"漏损"去报告，这是本题真正的难点所在，而不是简单套一个"母表减子表"的公式。下面会看到，数据本身就暴露出这三种情况都存在。

编码本身携带层级信息：例如 `416X → 41601X → 4160101T`，后一级编码总是以前一级的数字部分为前缀，末位字母 `X`（有下级）或 `T`（末端，但也可能仍有隐藏下级）作为标记。于是我写了一个"最长数字前缀匹配"算法自动重建父子树（`hierarchy_tree.csv`），而不是手工按列位置对应——因为同一个物理"母表"在不同分支里出现的列位置并不固定（有的母表出现在"一级"列，有的出现在"二级"列），必须按编码前缀而不是表格列号来判定父子关系。

```python
def find_parent(code):
    d = digits(code)
    best = None
    for c2 in code_list:
        d2 = digits(c2)
        if len(d2) < len(d) and d.startswith(d2):
            if best is None or len(code_digits[best]) < len(d2):

```
结果：93 个编码节点，**13 个根节点**（对应校园里 13 条相对独立的进水总管/区域），**17 个节点有子节点**（即"母表"），**91 个节点有真实物理水表读数**（另外 2 个是"无记录"的纯层级占位符，如 `402X`、`404X`），**76 个是没有下级的终端表**。

## 三、候选建模思路对比

### 问题①（功能区用水特征）：规则打标签 vs 无监督聚类

- **方案 A（规则/关键词打标签）**：水表名本身含有语义（"XXX第二学生宿舍""XXX第二食堂"），直接用关键词匹配打上"宿舍/食堂/教学科研楼/办公后勤/体育场馆/农业园林/基础设施/其他"八类标签，再统计各类的小时曲线、周末/工作日对比。
- **方案 B（无监督聚类）**：对每个水表的 24 小时用水曲线做标准化后 K-means 聚类，让数据自己"长出"类别，不依赖人工规则。

**取舍**：题目明确要求"给出校园内不同功能区（宿舍、教学楼、办公楼、食堂等）的用水特征"，也就是说功能区标签本身是已知的、有物理意义的分类目标，而不是要我们去发现未知的簇。水表名称本身就是可靠的"标签源"（真实数据里没有噪声或缺失的标签，91 个表全部可读出建筑名），用聚类反而是舍近求远、还可能把同一栋楼因为一次异常用水波动误聚到别的簇里，且聚类结果（"簇1""簇2"）缺乏物理解释力，最终还要人工回头对照建筑名字贴回"宿舍/食堂"标签——等于多绕一圈。所以选方案 A，把方案 B 作为"验证性备选"提及：如果关键词分类结果与后续小时曲线形状明显矛盾（比如某个被标成"宿舍"的建筑用水曲线长得完全不像宿舍），可以用聚类反过来核查分类是否有误——但在本次实际计算中没有出现这种矛盾，所以没有再跑一遍聚类。

### 问题②③④（母子表关系、漏损、定位）：这是本题的核心，考虑了三种方案

- **方案 A：水量平衡法（Water Balance / DMA 分区计量法）**。这是国际水协会（IWA）和国内供水行业通用的漏损核算方法的简化版：对树中每一个"母表"节点，比较"母表在时间窗口内的用水量"与"其直接子表用水量之和"，差值占比即为该分支的账面漏失率。
- **方案 B：机器学习异常检测**（如 Isolation Forest / 时间序列残差自编码器），不依赖层级树，直接对每个水表的时间序列找"异常点"。
- **方案 C：日尺度线性回归**，把"母表日用水量"作为因变量，"子表日用水量之和"作为自变量，拟合 y = a·x + b，用 R²/RMSE 衡量关系的稳健程度，作为方案 A 的补充和交叉验证。

**取舍**：题目原文明确要求"结合校区水表层级关系，建立水表数据之间的关系模型"——也就是必须用给定的树结构，而不是抛开树结构单独对每个水表做异常检测。方案 B 虽然通用性强，但缺乏"母表应该等于子表和"这个物理约束，也没有标签数据可以训练/验证，检测出来的"异常"无法直接解释成"漏损了多少方水"，不满足题目要"分析漏损"（要给出量化的漏损率）和"确定漏损位置"（要具体到树的某个分支）的要求。所以选方案 A 作为主模型，方案 C 作为方案 A 的误差量化补充（用回归的 R²/RMSE 交叉验证水量平衡法算出的漏失率是否"稳"，而不是只看一个点估计）。方案 B 未采用，但在"局限与展望"中提到可以作为后续改进方向（尤其是对树已经到底、无法再细分的叶子节点内部的定位）。

对问题④"定位暗漏"，进一步比较两种子方案：
- **树形逐层定位**：既然④要求"确定漏损位置"，天然的做法是把方案 A 的水量平衡不只做在根节点，而是**对树的每一层、每一个有子节点的母表都做一遍**，这样"漏损率异常"就能精确定位到具体是哪个分支（例如"区域4"分支）出问题，而不是笼统地说"校园漏水 13%"。
- **夜间最小流量法（Minimum Night Flow, MNF）**：这是供水行业检测"最后一公里"暗漏的经典方法——正常情况下，非居住类建筑在凌晨 2-4 点应几乎没有用水，如果某表在此时段仍有持续基流，大概率是持续性暗漏（水龙头滴漏、地下管道渗漏）。这个方法用于树已经到达叶子节点、无法再往下拆分子表的场景，是对"树形定位"分辨率不够时的补充手段。

两者互补使用：树形定位负责"是哪个分支"，夜间最小流量法负责"分支内部具体是哪栋建筑内部有持续暗漏迹象"。

### 问题⑤（维修决策）：静态回收期 vs 更复杂的运筹优化

- **方案 A：静态投资回收期（Payback Period）+ 分档决策规则**——用"年化漏损水量 × 水价"得到该分支漏损的经济价值，除以维修成本得到静态回收期，回收期短的优先修。
- **方案 B：0-1 背包/整数规划**——把"是否维修某分支"作为 0-1 决策变量，在总维修预算约束下最大化年化节水收益，是一个组合优化问题。

**取舍**：方案 B 理论上更完整，但需要一个"总预算"数字，而题目和附件都没有给出学校的维修预算规模，如果我编造一个预算再去优化，得出的"最优解"其实是被我编的预算数字决定的，容易营造"算法很高级但结果是假的"的错觉。方案 A 更朴素、更透明，每一步用的数字都能拆开看清楚（漏损率来自真实季度数据算出，水价和维修成本是**唯一**引入的示例性外部参数，并且在论文和本文件里都会反复标注"示例/说明性数值，非数据集给出"），得到的排序结论（先修哪个分支）与是否套 0-1 规划无关——因为在没有硬预算约束、维修成本量级远小于漏损年值时，两种方法的排序结果是一致的（回收期短即优先）。所以采用方案 A 为主，在"模型推广"部分提出：若学校提供了实际预算和真实维修单价，可以直接把本文算出的"各分支年化漏损量"代入方案 B 的 0-1 规划模型里，不需要重新做水量平衡计算。

## 四、关键假设及其论证

1. **假设：水表层级表中的父子编码关系代表真实的管网拓扑上下游关系（母表在上游，其编码前缀相同的子表在下游）。**
   论证：这是附件本身的设计意图（否则无法解释"表计编码"逐级细化、且母表水表名多带"+"后缀这种"总表"标记），而且实证上，401X、403X 两个"大"分支的母表-子表和吻合度极高（差值仅 0.3%~0.7%，见下文），如果拓扑关系是错的，不可能出现这么小的差值，说明假设在大多数分支上是成立的。

2. **假设：并非所有"母子表"关系都是"穷尽下游用户"的关系——有一部分是"总用量表 → 某个具体子用途的专用计量表"（如宿舍热泵热水表），这类子表天然只占母表用量的一小部分，差值不能算作漏损。**
   论证：实证发现，`4033501T→XXX3舍热泵热水` 这类关系差值高达 60%~90%，且子表名称明确写着"热泵热水"——这是宿舍楼里专门给热泵热水系统装的分表，不可能是宿舍总用水的唯一下游。若把这类差值也算进漏损率，会严重高估漏损，因此必须先按"该母表下的子表是否覆盖了母表名义上服务范围内的全部主要用户"这一标准，把 17 个母子关系分成"穷尽型"（8 个：401X/403X/405X/40133X/40134X/40135X/40335X/40337X，多子表、覆盖一个"区域"或"栋群"）和"局部专用型"（其余，多为单一子表指向具体设备用途），只对前者做漏损核算。

3. **假设：短期（一个季度）内管网物理结构和口径不发生改变，漏损率的变化主要反映用水量变化和真实渗漏变化，而非管网改造。**
   论证：题目只给了分季度数据、且没有任何管网改造记录，只能假设季度内结构稳定；这个假设也用敏感性分析做了检验（见下文，把季度拆成前后两半看漏损率是否稳定）。

4. **假设：非居住类建筑（教学楼、体育馆等）在凌晨 2-4 点应无正常人为用水，此时段的持续非零流量可判定为潜在暗漏信号。**
   论证：这是国际通行的"夜间最小流量法"的基本假设，符合校园作息常识（凌晨教学楼、体育馆均已关闭）；但对宿舍类建筑不适用（凌晨仍有人在寝室用水是正常现象），所以在计算里专门把"宿舍""办公/后勤/接待（含旅馆性质）"排除在这个判据之外。

5. **假设：水价、维修成本使用示例性数值（非居民/公共机构水价约 4.9 元/m³，维修成本按严重程度分档 5000/15000/30000 元），仅用于展示决策模型的计算逻辑，不代表该校真实价格。**
   论证：数据集本身不含价格信息，题目原文写"结合……你了解的水价和维修成本"，本身就是要求引入外部经验值；论文中会显著标注这些数字的性质，避免读者误以为是从数据中算出来的。

## 六、遇到的问题与调整（汇总）

1. **发现问题**：字符串日期比较导致"看起来"季度数据只到3月9日，实际是字符串字典序比较的假象。
   **调整**：所有涉及日期范围/排序的地方，先 `pd.to_datetime` 转换再比较，不能相信字符串排序结果。

2. **发现问题**：如果不区分"穷尽型"和"局部专用型"母子关系，直接对全部 17 个母子关系算差值占比，会把热泵热水表这类"专表"的差值（60%~90%）误当作管网漏损，严重失真。
   **调整**：引入"该母表的子表是否是覆盖多个独立下游用户的区域/楼群式子表"这一判据，人工核查子表名称语义，把 17 个关系分成 8 个"穷尽型"（纳入漏损核算）和其余"局部专用型"（不纳入，单独说明其性质），只对前者报告漏损率。

3. **发现问题**：`区域1（西）(40133X)` 出现子表和超过母表的负值异常。
   **调整**：先排查是否为数据覆盖时间不一致的假象（逐表核对起止时间和条数），确认六个表时间口径完全一致后，判定这是真实的计量/拓扑异常而非负漏损，单独作为"表具需现场核查"类问题报告，不并入漏损统计的平均值（或在报告加权平均时明确标注剔除前后两个数值）。

4. **发现问题**：粗略的"前后两半"敏感性分析显示 40337X 漏损率在"上升"，一开始据此写下"这是恶化中的新增漏损"的结论。
   **调整**：进一步用逐日 CUSUM 变点检测复核，发现实际上不存在单一突变点、差值序列在高位反复波动，据此推翻了"新增漏损、正在恶化"的初步判断，改为"持续性结构缺口，非季度内新发"的更保守、更贴合数据的结论。这是本次求解过程中最典型的一次"计算结果推翻直觉"的调整，完整保留在正文。

5. **发现问题**：功能区分类的"其他商业生活服务"兜底类别偏大（28/76个叶子表），说明仅凭几个关键词分类颗粒度不够细。
   **调整**：没有强行把这些杂项用户塞进"宿舍/食堂/教学楼/办公楼"四大类（那样会污染这四类的特征统计），而是保留一个诚实的"其他"类别，在论文里明确说明这是分类方法的局限，重点结论只针对题目点名的四类功能区。

6. **发现问题**：题目和附件均未给出水价、维修成本，若强行"用数据算出"这两个数字，就是编造。
   **调整**：明确使用外部常识性示例值（并在正文、摘要、免责声明多处标注"示例/说明性数值"），把模型的核心贡献放在"排序逻辑和回收期计算方法"上，而不是放在具体的元/立方米数字上，这样即使学校的真实水价不同，模型和排序方法依然成立，只需要替换输入参数重新算一遍。

## 七、自我批判：局限与如果重来会怎么改进

1. **只用了一季度数据**。附件里其实还有二、三、四季度数据（各约 23~26MB），本次受时间预算限制只处理了一季度。如果重来，应该把 5.4 节的敏感性分析和 CUSUM 变点检测扩展到全年四个季度，才能更可靠地区分"季节性波动"（如寒暑假）和"真实新增漏损"，现在的结论（尤其是"40337X 是结构性缺口而非新增漏损"）只在一季度数据支持下成立，样本量偏薄。

2. **"穷尽型 vs 局部专用型"的判定是人工规则（看子表数量和名称语义），不是算法自动识别的**。如果子节点数量更多、名称语义更模糊，这个规则会失效。更严谨的做法是引入"子表口径之和是否接近母表口径"这类工程量纲信息做交叉验证（水表层级表里有"口径"字段，本次没有用上，是一个明显的遗漏），或者用回归斜率是否显著偏离1、R²是否够高作为自动判据的一部分。

3. **夜间最小流量法的"异常"判据（夜/日流量比>1）偏简单，没有做统计显著性检验**，也没有排除偶发性事件（比如某晚保洁大扫除用水）造成的单夜异常——只用了整季度平均，抹平了这种偶发性，但同时也可能漏掉真正"只发生几天就被发现修复"的短期暗漏。更完整的做法应该逐夜计算、用滑动窗口+假设检验判断"持续性"而不是只看季度平均比值。

4. **维修决策模型是静态的、单期的（一次性判断修不修）**，没有考虑管网漏损会随时间推移而恶化（未修的漏损可能今天10%明天变15%），也没有考虑维修资源（人工队伍数量）的排队约束。更完整的模型应该是一个多期动态规划或排队论模型，本次因为价格、成本、恶化速率等参数都缺乏真实数据支撑，做多期模型的"精细结果"意义不大（本质是在编造的数字上做更复杂的运算），所以有意选择了更朴素、更透明的静态模型。

5. **功能区分类基于关键词，人工核对样本量有限**，93个表里我抽查了十几个但没有逐一读原始水表名称核对分类是否全部合理，存在个别误分类的可能（例如"其他商业生活服务"类里可能混入个别应属于"办公"或"教学"的表）。

6. **水价、维修成本是本文唯一"非数据驱动"的输入**，任何基于这两个数字的绝对金额结论（如"回收期0.3年"）都只是方法论演示，不能作为学校真实决策依据，这一点在论文摘要、正文、免责声明里都反复强调，是本文档最需要读者注意的局限。
