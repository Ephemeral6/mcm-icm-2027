## 1. 题目理解与拆解

在读完 `/Users/ephemeral/美赛/08-历年真题/2022/2022_MCM_Problem_C.md` 后，我把题目拆成以下几个必须回答的技术问题：

1. **状态是什么**：每天的组合是 `[C, G, B]`（现金、黄金盎司数、比特币数），初始 `[1000, 0, 0]`，起点 2016-09-11，终点 2021-09-10（5年，1826个日历日）。
2. **决策变量是什么**：每天对黄金、比特币各自做 Buy / Hold / Sell 三选一（可以理解为改变持仓权重）。
3. **信息约束（关键！）**：策略"只能用截止当天为止的历史价格"——这是一个**因果性（causality）**约束，是本题最容易被判定"作弊"的地方：任何用到未来价格的模型（哪怕只是提前知道当天收盘价来决定当天早上要不要买）都是不合规的。很多网上能搜到的"高分论文"实际上用未来数据回测调参，这是本题的一个常见陷阱，我在建模时特别注意每个信号都必须是"滞后可得"的。
4. **摩擦约束**：买卖手续费 α_gold=1%，α_bitcoin=2%，双向都收（买入和卖出各收一次），持有免费。
5. **交易日历约束**：比特币每天可交易，黄金只有在 LBMA 报价的工作日才能交易——这意味着模型要处理"黄金当天不可交易"这个离散约束，很多论文会忽略或简化错。
6. **优化目标**：终值 V(T)=C+G·P_g(T)+B·P_b(T) 最大化。
7. **还需要**：证明模型是"最优"的证据、对交易成本的敏感性分析、一份给交易员看的备忘录（≤2页）。

## 2. 候选建模思路对比

在正式建模前，我列了 4 个可能的方向，逐一评估取舍（这是"不要n板斧、先想清楚机理"的具体体现）：

### 方案 A：纯技术指标规则（如双均线交叉 / RSI 阈值）
- 优点：直观、透明、天然满足因果性（只用历史价格算出的均线一定不偷看未来）、计算量极小。
- 缺点：本质是一个二元 all-in/all-out 开关，**没有显式地把"预期收益 vs. 交易成本 vs. 风险"三者放在一起权衡**——均线一有交叉就无脑满仓进出，在震荡市（尤其比特币这种高波动资产）里会被反复"打脸"（whipsaw），产生大量无意义的手续费损耗。
- 后来我把这个方案实现出来当作"对照组"（`naive_momentum` 基准），实测验证了这个担心：在我的示例实验里，均线交叉策略跑赢"什么都不做（现金）"的比例只有 13%，说明过度交易的问题是真实存在的，不是臆测。

### 方案 B：已知价格转移模型下的动态规划 / 随机最优控制（Bellman 方程）
- 想法：把黄金、比特币价格建模成某个随机过程（例如 GBM 或马尔可夫链），估计出转移概率/参数后，用逆向动态规划求出真正的"最优"买卖阈值策略——这是 Merton 型资产配置问题在离散时间、带交易成本下的标准做法。
- 致命问题：
  1. 要在"只用截止当天数据"的约束下做，就必须先用历史数据估计出一个转移模型，而比特币 2016-2021 年的波动率状态变化极大（早期波动率、2017年泡沫、2018年熊市、2020年后又完全不同），5年数据去估计一个"稳定"的转移模型是病态问题，模型误设风险很高。
  2. 更关键的：如果直接在**已实现的历史价格路径**上做逆向 DP（很多论文就是这么干的，效果好看），这本质上是在用"未来已经发生的价格"去算"过去应该怎么做"，是不折不扣的信息穿越，**根本不能当作可执行的交易策略**，只能当作事后的"完美信息上界"用来做对照（我在最终模型里正是这么用它的——见下文 oracle 部分）。

我把 B 方案改造成了论文里的"非因果 DP 上界（oracle）"，作为评估基准，而不是作为实际可执行策略。

### 方案 C：机器学习预测（如 LSTM）+ 阈值交易
- 想法：训练一个序列模型预测下一日收益，预测值超过覆盖交易成本的阈值就交易。
- 缺点：
  1. 5年、且黄金还有约500天不交易，有效训练样本本就不算多，深度序列模型很容易过拟合到训练区间的特定行情；
  2. 预测的不确定性/置信区间很难和"要不要付1-2%手续费"这个决策规则干净地对接，对交易员而言"黑箱预测数字"远不如"均线动量 + 协方差 + 效用函数"这套东西可解释；
  3. 没有对接经典的"比例交易成本下的最优控制"理论（下面选用的方案就是直接对接这套理论），显得"为了用模型而用模型"。

### 方案 D（最终选用）：动量信号驱动的单期均值-方差目标权重 + 成本感知"不交易区间"（no-trade band）
把 A 的"透明因果信号"和 B 的"风险-收益-成本联合权衡"结合起来：
- 用纯因果的双 EMA 动量信号 + 滚动协方差矩阵，构造每天的"预期收益、风险"估计；
- 每天求解一个只有 3 个变量（现金/黄金/比特币权重）的均值-方差目标配置（类似单期 Markowitz 问题）；
- **不是每天都调仓到目标权重**，而是只有当"调仓带来的效用提升 > 调仓要付出的手续费"时才交易，否则 Hold——这正是 Constantinides (1986)、Davis & Norman (1990) 这类"比例交易成本下最优组合存在一个不交易区间"理论的简化离散版本。

选择 D 的理由：
1. 严格因果（所有信号只用 t 时刻及之前的价格）；
2. 显式地把交易成本作为决策规则的一部分（而不只是事后扣个手续费），直接对应题目"Determine how sensitive the strategy is to transaction costs"的要求——成本参数天然地出现在"要不要交易"这个不等式里，做敏感性分析非常自然；
3. 可以对交易员解释："模型认为现在换仓能带来的好处（收益预期提升、风险下降）超过手续费，所以换"——这是一句人话，不是"神经网络觉得该买"；
4. 计算量小（每天一个 3 变量二次规划，SLSQP 求解，1826天全程几秒内跑完），符合"用截止当天数据实时决策"的实际交易场景。

## 3. 关键假设及其论证

| 假设 | 论证 |
|---|---|
| 用双 EMA（短10日/长40日）动量比 `EMA_short/EMA_long - 1` 作为预期收益代理 | 纯因果、计算简单、经典技术分析里"金叉死叉"思路的连续化版本，避免了二元开关的僵硬性 |
| 用滚动30日窗口的对数收益协方差矩阵作为风险估计 | 同样因果；30天窗口是短期波动率和统计稳定性之间的折中（窗口太短噪声大，太长跟不上比特币这种波动率会突变的资产） |
| 单期（myopic）均值-方差效用，而非多期动态规划 | 多期最优在存在比例交易成本时需要求解自由边界偏微分方程（Davis-Norman），解析/数值复杂度对一个"给交易员用"的模型而言过高；单期近似是业界常见的实用简化，代价是可能不是全局最优（已在"自我批判"部分说明） |
| 不允许卖空（权重 ≥ 0） | 题目背景是一个普通交易者的实盘账户，没有提到可以卖空；现金、黄金、比特币权重非负且和为1，符合真实约束 |
| 黄金不可交易日，黄金持仓量强制不变，只允许现金与比特币之间调仓 | 直接对应题目"gold is only traded on days the market is open"的硬约束，在优化问题里通过一条等式约束 `w_gold = 上一交易日权重` 实现 |
| 交易执行"自融资"（self-financing）：现金不能变负 | 这是最基础的会计恒等式，属于常识性假设，但在实现中我一开始漏掉了（见下"遇到的问题"），后来专门加了修正 |

### 4.3 第一次实现策略引擎，发现资金变负的 bug
第一版执行逻辑：算出目标权重对应的目标市值，直接按目标市值算买卖金额、扣手续费。跑出来：
```
        date       C         G    B            V    action
8  2016-09-19   -10.0  0.737548  0.0   990.000000  BUY-GOLD
...
1825 2021-09-10  -0.2924  0.017176  1.414526e-15  30.390120   HOLD
```
现金 `C` 出现了负数！这是一个真实的、必须修的 bug：原因是"用目标市值反推交易额"这个写法，没有考虑到"买入的手续费是额外付出的"，如果均值-方差最优解让你 100% 买入某个风险资产（权重设为1，现金权重设为0），实际需要的现金 = 目标市值 ×(1+手续费)，超过账上现金，就会把现金"透支"成负数。

**修复**：改成"先执行卖出（换回现金），再看剩余现金能覆盖多少买入需求，覆盖不了就按比例 κ 缩减买入量"，保证交易永远是自融资的（现金不会变负）。修完之后重新跑：
```
8  2016-09-19     0.0  0.730245  0.0   990.09901  BUY-GOLD
...
1825 2021-09-10  0.0  0.018919  1.560014e-15  33.795243   HOLD
```
现金恒 ≥ 0，问题解决。这个 bug 和修复过程被我保留在代码注释里（`full_experiment.py` 里能看到），也如实写在这里——这是"真做了一遍建模"里非常真实的一步：**很多模型的"理论最优权重"和"account 层面能否实际执行"是两回事**，这个教训本身就值得写进论文的 Model Development 部分。

### 4.5 构造非因果的"事后最优"DP 上界作为参照
为了满足题目"Present evidence that your model provides the best strategy"的要求，需要一个可信的比较基准。最开始我写了一个"贪心式"的完美预知 oracle（每天看下一天涨得最多的资产满仓买入），结果这个 oracle 在有些路径上竟然跑输 50/50 买入持有策略——这是不合理的，因为"完美信息"理论上不可能比任何因果策略差。

**问题排查**：贪心 oracle 每天都可能换仓（哪怕次日涨幅很小），换仓就要付手续费，长期高频换仓的手续费侵蚀掉了大部分"完美预知"的优势，这不是一个真正的最优策略，只是一个次优的启发式。

**修正**：改用严格的逆向动态规划（backward induction），状态空间是{现金, 黄金, 比特币}三个离散状态（在确定性/事后已知价格的前提下，最优解一定是集中持有单一资产，不存在分散化收益，这是可以严格证明的），递推关系：
```
g(t, s) = max_{s'} [switch_cost(s→s') × price_return(s', t→t+1) × g(t+1, s')]
```
这是一个真正的（针对"单资产满仓切换"这类策略中）最优上界。重新跑出的结果：
```
oracle_upper_bound    mean=$92,449,737.3  ...  share_beats_cash=100%
```
数量级从"几千美元"跳到了"9千万美元"——**这也是一个如实记录的、初看意外的现象**：因为在我们设定的日波动率下（比特币日波动率约2.9%），只要"round-trip 手续费（约3%）小于典型的单日价格波动"，一个知道每天涨跌方向的"上帝视角"策略就能靠天天赚"日内噪声"呈指数复利增长，1826天复利下来就是天文数字。这不是 bug，是这类"事后已知每日方向"DP 上界的已知病理特征（在文献里也有类似讨论），**它说明这个 oracle 数值本身不是一个有实际意义的目标**，只能定性地说明"若能预知每天方向，理论收益上限极高"，我在论文的 Solution 和 Strengths/Weaknesses 部分都对此做了明确说明，避免误导读者以为"我们的策略离最优差了10万倍"这种没有意义的比较。

```
3.0     999.94              516.49               40.40
5.0    1028.78              511.56               41.27
8.0     943.53              488.73               41.53
15.0    870.52              503.51               41.73
```
**清晰、单调、符合理论预期的结论**：随着交易成本从0倍上升到4倍基准值，模型的平均交易次数从122.3次单调下降到8.3次——这正是"不交易区间随成本变宽"这一经典结论的数值验证。终值（wealth）对成本的响应则不是单调的（0倍成本时均值最高但中位数并非最高，且0.5倍成本时均值反而低于1倍成本），说明"少交易不一定赚更多、多交易不一定亏更多"，交易成本对最终财富的影响是路径依赖、非单调的，只有"交易频率"这一个维度呈现出干净的单调关系——这个区分（频率单调、终值不单调）本身就是一个值得写进论文的发现。γ 扫描则显示：交易频率对 γ 不敏感（40~42次附近小幅波动），终值随 γ 变化也没有单一方向的趋势，说明在当前的信号强度量级下，风险厌恶系数在测试范围内主要影响"敢不敢重仓"，而不是"要不要交易"（是否交易主要由动量信号与成本的相对大小决定）。

## 5. 遇到的问题与调整（如实记录，含"发现问题→定位原因→修复→复核"的完整链条）

1. **现金透支（负值）**：见 4.3，已修复（自融资执行逻辑）。
2. **贪心 oracle 不合理地跑输因果策略**：见 4.5，已用严格逆向 DP 替换。
3. **不交易带的边界条件 bug（`cost_frac > 1e-9`）**：最初的交易触发条件写成 `(U_tgt - U_pre) > cost_frac AND cost_frac > 1e-9`，本意是防止"成本几乎为零、微小数值噪声也被当成交易"，结果在做"交易成本=0"这个敏感性分析点时，这个 AND 条件让模型永远不交易（因为 cost_frac 恒为0，`0 > 1e-9` 恒假），导致"零成本"这个本该交易最频繁的极端情形，却输出0次交易——这是在看敏感性分析结果时发现的异常（当时第一次跑出来 `cost_multiplier=0.0` 那一行 `mean_n_trades=0`，明显不符合"成本越低应该交易越频繁"的直觉，才回头去查条件判断逻辑）。
4. **修复#3之后引入的新问题**：去掉 `cost_frac > 1e-9` 这个 AND 条件后，蒙特卡洛结果里模型的平均交易次数从约39次翻倍到约73次，而终值几乎不变——怀疑是求解器 SLSQP 在最优点附近的数值噪声（比如目标权重和当前权重只差 1e-10）被放行执行了，属于"看起来是交易、实际上是不该发生的数值抖动"。**修复方式**：把"是否有意义的交易"判据从"成本是否大于极小值"改成直接判断"目标权重变化量 `dw_max` 是否超过 1e-4"，这样即使成本恰好为0，只要权重变化是有实际意义的（>0.01%），依然会被识别为一次真实交易；同时数值噪声级别的权重抖动（<1e-4）会被正确过滤掉。
   - 修复后重新验证：单独调用 `run_strategy` 检查 seed=3 路径，交易次数回落到 41 次（和最初未引入 bug 时的量级一致），确认修复生效。
5. **磁盘上残留了修复前的旧结果文件**：见 4.6。修完 bug 4 之后，第一次重新运行 `sensitivity.py`/`monte_carlo.py` 是通过链式后台命令触发的，结果 `monte_carlo_results.csv`（后写入）反映了修复后的正确行为，但 `sensitivity_cost.csv`/`sensitivity_gamma.csv`（先写入）里的交易次数数字和修复前完全一致，怀疑是命令链发起时机与代码保存时机之间存在竞态，或是旧后台任务写入延迟覆盖了新结果。发现方式：拿同一份种子集在两份文件里的数字互相对不上（monte_carlo 里 seed=3 是41次交易，但 sensitivity 汇总的均值明显偏高）。处理方式：不去猜测具体原因，而是清空 `__pycache__` 和所有中间日志、用阻塞式命令重新完整跑一遍并用 `SENSITIVITY_DONE_MARKER` 标记确认真正跑完，再做一次独立单种子交叉验证，才把结果写进论文。这提醒我：**涉及后台/并发执行时，"文件存在""时间戳晚于代码修改"都不能保证内容是最新的，必须有一个独立的交叉验证步骤**。

## 6. 自我批判：局限与如果重来会怎么改

1. **没有真实数据**：这是本次任务最大的局限，一切"结果"都只是示例参数下的说明性数值，不能作为真实策略性能的证据。如果拿到真实 LBMA/BCHAIN 数据，第一步应该先看真实的收益率分布、波动率聚集特征，再决定 EMA 窗口、协方差窗口这些超参数是否合适（目前是凭经验设的10/40/30天，没有做真正的调参）。
2. **单期（myopic）均值方差不是多期最优**：真正严谨的"比例交易成本下的最优控制"需要解自由边界问题（不交易区间的边界是价格/信号的函数，而不是我们这里简化的"一步效用增益 vs. 成本"启发式判据）。这是刻意的简化，论文里也说明了这一点，但如果时间允许，值得对比一个真正的多期动态规划或强化学习方案，看看差距有多大。
3. **γ（风险厌恶）和窗口长度是人为设定的**，没有做样本内外的正式调参/交叉验证，这在真实数据下是必须补的一步。
4. **oracle 上界的数量级问题**：如 4.5 所述，逐日完美预知的 DP 上界会因为"日内噪声套利"而炸到不合理的数量级，这个指标对"评估我们的模型有多接近最优"帮助有限，如果重做，会考虑加一个更贴近现实的上界（比如"每周最多调仓一次的完美预知策略"），会更有参考意义。
5. **50/50 买入持有基准跑赢我们模型的比例偏高（87%）**，说明当前的风险厌恶参数/信号设计整体偏保守，牺牲了不少上涨趋势中的收益。如果这是一次真实参赛，我会把这一点作为改进方向而不是藏起来——先分析是"仓位上限太保守"还是"动量信号滞后太严重"，再针对性调整。
