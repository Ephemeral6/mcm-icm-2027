# 04 三人分工模式

> 本文件内容以社区实战经验为主（**【经验·未验证】**，来源：知乎高赞问答、CSDN 获奖者复盘），分工建议部分是针对你们队情况的综合建议。
> 官方相关事实：队伍上限 3 人、历史最佳队伍均为 3 人（已验证）。

## 1. 经典分工：建模手 / 编程手 / 写作手

| 角色 | 职责 | 备赛主攻 |
|---|---|---|
| 建模手 | 把实际问题数学化，选择与构建模型，主导论文思路 | 常用模型清单（见 05）、读 O 奖论文的建模章节 |
| 编程手 | 实现模型求解、数据处理、结果检验、出图 | Python 数据栈、各模型的模板代码（输入/输出/适用场景） |
| 写作手 | 将模型与结果组织成完整英文论文，摘要、排版、图表美化 | LaTeX 模板、O 奖论文行文结构、学术英语 |

## 2. 对"严格分工"的批判——不要照搬

多位获奖者指出，"建模手只建模、编程手只编程、写作手只写作"是**虚假分工**，实战中的典型翻车场景：

- 建模手搭好精妙复杂的模型，编程手回一句"这实现不了"，前两天白干；
- 写作手前期无事可做、后期独自深夜赶稿，论文质量崩盘；
- 三人对模型理解不一致，论文前后矛盾。

改良共识：
- **人人都要懂建模**——推荐结构：队长精通建模+编程+写作并负责协调；队员 1 主攻建模+编程+可视化；队员 2 主攻建模+写作+图形美化。
- 开赛头几个小时**三人共同**读全部题目、查文献、热烈讨论，全员吃透选定模型后再分头执行、滚动协作。
- 写作从第二天就开始，与建模求解并行，绝不留到最后一天。

## 3. AI 时代的分工演变

- **编程手优先学 Python 而非 MATLAB**：获奖者直言"AI 辅助写代码的效率 Python >> MATLAB"。
- 有 2024 年 A 题参赛队公开了赛中 GPT-4 工作流：让 AI 扮演"美赛指导老师"→ 梳理题意 → 给建模思路 → 用数学语言写出模型 → 生成对应 Python 代码 → Debug。这类流程现在已是普遍操作，**差异化在于谁能把 AI 输出验证好、解释好**（评委反感黑箱，见 06）。
- AI 压缩了编程与初稿写作的工时，队伍的稀缺能力变成：题意理解、模型假设的论证、结果解读、图表设计、摘要打磨——分工应向这些环节倾斜人力。
- 合规要求（见 02）：AI 交互全程留痕是一个新增的"岗位职责"，必须落实到人。

## 4. 你们队的分工建议（综合建议）

你的画像：AI 工具熟练、数理编程一般 → 你适合做 **"AI 工程师 + 编程/可视化手"**：

| 成员 | 角色 | 具体职责 |
|---|---|---|
| 你 | AI 工程师 + 编程手 | 搭建全队 AI 工作流（提示词库、交互留痕文档）、AI 辅助写代码与出图、维护 AI Use Report 素材 |
| 队友 A（三人中数学最好的） | 建模手 + 队长 | 模型选择与假设论证、把关 AI 建模建议、协调进度 |
| 队友 B（三人中英文最好的） | 写作手 | LaTeX 排版、论文行文、摘要主笔、图表美化 |

前提动作（8 月前）：三人互相摸底数学/编程/英文水平后正式定岗；每人补齐"建模通识"（都能看懂常用模型清单，见 05）。

## 5. 磨合训练方法（社区共识）

1. **至少一次完整模拟赛**：选一道近年真题（建议 2024/2025 年 C 题），严格 96 小时全流程产出完整英文论文——目的是跑通协作与工时分配，不是拿高分。
2. 模拟赛以 **O 奖论文为对标模仿对象**；获奖者警告：不要试图复现 O 奖以下论文的思路（很多是"口水建模"，图表与陈述对不上，复现极难且误导）。
3. 可先参加 11 月的"小美赛"（认证杯等）作为带奖模拟。
4. 模拟后复盘三件事：时间去哪了、哪个环节卡壳、AI 用在哪最省时。
